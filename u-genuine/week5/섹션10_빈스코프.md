### 빈의 생명주기

스프링 컨테이너가 생성되면 빈들도 함께 생성되고 관리되다가

스프링이 종료되기 직전에 빈들도 함께 종료된다

**Why?**

- 스프링 빈은 기본적으로 **싱글톤 스코프**로 생성되기 때문
- 싱글톤 스코프는 **스프링 컨테이너의 수명과 동일하게 관리됨**

> **싱글톤 스코프**의 특징
> - 컨테이너의 시작부터 종료까지 하나의 빈이 싱글톤으로 관리됨
> - 가장 생명 주기가 길다

---

### **프로토타입 스코프**

- 스프링 컨테이너가 **빈의 생성과 의존관계 주입까지만 관리**
- 이후의 생명주기는 컨테이너가 관여하지 않음
- 따라서 종료 메서드도 호출되지 않는다 → 클라이언트에서 직접 종료메서드를 호출해야 함

### 프로토타입 빈 요청 과정

1. 클라이언트가 프로토타입 빈 요청
2. 스프링 컨테이너는 새로운 빈 생성 + 의존성 주입 + 초기화
3. 생성한 빈을 클라이언트에게 반환하고 컨테이너는 더 이상 빈을 관리하지 않는다
4. 이후에 같은 요청이 와도, 항상 **새로운** 프로토타입 빈을 생성해서 반환함 → 반환 후엔 컨테이너에 관리되고있는 프로토타입 빈이 없기 때문에 기존의 빈을 줄 방법이 없다

---

### 싱글톤 스코프 vs 프로토타입 스코프

**싱글톤 스코프 빈 테스트**

```java
@Scope("singleton")
static class SingletonBean{
	
		@PostConstruct
		public void init() {
			System.out.println("SingletonBean.init");
		}
		
		@PreDestroy
		public void destroy() {
			System.out.println("SingletonBean.destroy");
		}
}
```

```java
@Test
ㅇvoid singletonBeanFind() {
		AnnotationConfigApplicationContext ac = new AnnotationConfigApplicationContext(
			SingletonBean.class);

		SingletonBean bean1 = ac.getBean(SingletonBean.class); // 싱글톤 빈 생성
		SingletonBean bean2 = ac.getBean(SingletonBean.class); // 이미 생성된 싱글톤 빈 반환
	
		assertThat(bean1).isSameAs(bean2); // 같은 인스턴스
		ac.close();
	}
```

**실행 결과**

```bash
SingletonBean.init
SingletonBean.destroy
```
- 빈 **초기화** 메서드는 **1번 실행**
- bean1과 bean2는 **같은 객체**
- 종료 메서드가 자동으로 실행됨 → 스프링 컨테이너에서 싱글톤 빈의 소멸까지 관리하기 때문


**프로토타입 스코프 빈 테스트**

```java
@Scope("prototype")
static class PrototypeBean {

		@PostConstruct
		public void init() {
			System.out.println("PrototypeBean.init");
		}

		@PreDestroy
		public void destroy() {
			System.out.println("PrototypeBean.destroy");
		}
}
```

```java
@Test
void prototypeBeanFind() {
		AnnotationConfigApplicationContext ac = new AnnotationConfigApplicationContext(
			PrototypeBean.class);
			
		PrototypeBean prototypeBean1 = ac.getBean(PrototypeBean.class);
		PrototypeBean prototypeBean2 = ac.getBean(PrototypeBean.class); // 새로운 프로토타입 빈이 생성됨
		
		System.out.println("prototypeBean1 = " + prototypeBean1);
		System.out.println("prototypeBean2 = " + prototypeBean2);
		
		assertThat(prototypeBean1).isNotSameAs(prototypeBean2); // 서로 다른 객체
		
		// destroy 메서드 호출되지 않음
		ac.close();
	}	
}
```

**실행 결과**

```bash
PrototypeBean.init
PrototypeBean.init
prototypeBean1 = hello.core.scope.PrototypeTest$PrototypeBean@37b70343
prototypeBean2 = hello.core.scope.PrototypeTest$PrototypeBean@306851c7
```

- init 메서드가 각 객체 호출 후에 실행됨 (2번)
- 두 객체는 다른 객체
- 종료 메서드가 자동으로 실행되지 않았음 → 빈의 소멸은 스프링 컨테이너가 관리하지 않음

---
그럼 빈을 어떻게 종료하지?  
→ 클라이언트가 destroy() 직접 호출

```java
@Test
void prototypeBeanFind() {
		// ..
		// 클라이언트에서 직접 종료 메서드 호출
		prototypeBean1.destroy();
		prototypeBean2.destroy();
		
		ac.close();
}
```

**실행 결과**

```bash
PrototypeBean.init
PrototypeBean.init
prototypeBean1 = hello.core.scope.PrototypeTest$PrototypeBean@37b70343
prototypeBean2 = hello.core.scope.PrototypeTest$PrototypeBean@306851c7
**PrototypeBean.destroy
PrototypeBean.destroy**
```

- 클라이언트 코드에서 직접 종료 메서드를 호출
- destroy 메서드가 각각 호출되었다

### 싱글톤 빈이 프로토타입 빈을 의존성 주입 받으면?

```java
@Scope("singleton")
static class ClientBean { // 싱글톤 스코프 빈
		private final PrototypeBean prototypeBean; // 프로토타입 빈 생성 후 의존성 주입
	
		@Autowired
		public ClientBean(PrototypeBean prototypeBean) {
			this.prototypeBean = prototypeBean; // 생성 시점에 주입
		}
	
		public int logic() { 
			prototypeBean.addCount();
			return prototypeBean.getCount(); 
		}
}
```

```java
@Test
void singletonClientUsePrototype() {
		AnnotationConfigApplicationContext ac = new AnnotationConfigApplicationContext(ClientBean.class, PrototypeBean.class);
	
		ClientBean clientBean1 = ac.getBean(ClientBean.class);
		int count1 = clientBean1.logic(); 
		assertThat(count1).isEqualTo(1);
	
		ClientBean clientBean2 = ac.getBean(ClientBean.class);
		int count2 = clientBean2.logic();
		assertThat(count2).isEqualTo(2); // 같은 프로토타입 빈 사용 중
}
```
- 프로토타입 빈은 **ClientBean 내부에 보관**되고 싱글톤 빈과 함께 계속 유지
- 이미 생성되어 주입된 인스턴스를 공유하게 됨
- 프로토타입 빈은 매번 새로운 객체를 생성받는 용도로 쓰이기 때문에 옳지 않은 쓰임

---
**해결 방법: ObjectProvider 사용**

```java
	@Scope("singleton")
	static class ClientBean {

		@Autowired
		private ObjectProvider<PrototypeBean> prototypeBeanProvider;

		public int logic() {
			// ObjectProvider가 스프링 컨테이너에서 PrototypeBean 빈을 찾아줌
			// 매번 새로 요청
			PrototypeBean prototypeBean = prototypeBeanProvider.getObject();
			prototypeBean.addCount();
			return prototypeBean.getCount();
		}
	}
```

ObjectProvider를 이용하면 스프링 컨테이너에서 매번 새로운 프로토타입 빈을 찾아서 꺼내준다

- 스프링 컨테이너는 프로토타입 빈을 반환한 후에 더이상 관리하지 않기 때문에
- 매번 찾을때마다 새로 생성해서 반환
- ApplicationContext 전체 주입보다 가볍고, 의존도가 낮다

또는 자바의 Provider

- 자바 표준 API로, 스프링이 아닌 다른 프레임워크에서도 사용 가능
- 기능은 유사하지만, 스프링의 ObjectProvider가 더 편리한 기능이 많다

### 웹 스코프 (Request, Session 등)

- 웹 환경에서 사용 가능
- 프로토타입과 다르게 스프링이 스코프 종료 시점까지 관리
- 따라서 종료 메서드가 정상적으로 호출됨

**request 스코프**

- 하나의 HTTP 요쳥 당 하나의 request 스코프가 생성됨
- 요청이 들어와서 응답이 나갈 때까지가 생명 주기

**request 스코프 빈인 MyLogger 코드**

```java
@Component
@Scope(value = "request")
public class MyLogger {

	private String uuid;
	private String requestURL;

	public void setRequestURL(String requestURL) {
		this.requestURL = requestURL;
	}

	public void log(String message) {
		System.out.println("[" + uuid + "] " + "[" + requestURL + "] " + message);
	}

	@PostConstruct
	public void init() {
		uuid = UUID.randomUUID().toString();
		System.out.println("[" + uuid + "] request scope bean create: " + this);
	}

	@PreDestroy
	public void close() {
		System.out.println("[" + uuid + "] request scope bean close: " + this);
	}
}
```

- 로그 출력을 위한 클래스
- request 스코프이므로, HTTP 요청마다 새로 생성되고 응답 후 소멸됨
- requestURL은 빈 생성 시점에는 알 수 없어 외부에서 주입 필요
---
#### 문제 상황

```java
@Controller
@RequiredArgsConstructor
public class LogDemoController {

	private final LogDemoService logDemoService;
	private final MyLogger myLogger;

	@RequestMapping("log-demo")
	@ResponseBody
	public String logDemo(HttpServletRequest request) {
		String requestURL = request.getRequestURL().toString();
		myLogger.setRequestURL(requestURL);

		myLogger.log("controller test");
		logDemoService.logic("testId");
		return "OK";
	}
}
```

- 컨트롤러는 애플리케이션 시작 시 빈으로 등록이 되면서 의존관계 주입이 이뤄짐
- MyLogger는 request 스코프 빈이기 때문에 HTTP 요청이 들어오지 않으면 생성되지 않음 → 오류 발생

#### 해결 방법 1. ObjectProvider 사용

```java
@Controller
@RequiredArgsConstructor
public class LogDemoController {

	private final LogDemoService logDemoService;
	private final ObjectProvider<MyLogger> myLoggerProvider;

	@RequestMapping("log-demo")
	@ResponseBody
	public String logDemo(HttpServletRequest request) {
		String requestURL = request.getRequestURL().toString();
		MyLogger myLogger = myLoggerProvider.getObject();
		myLogger.setRequestURL(requestURL);

		myLogger.log("controller test");
		logDemoService.logic("testId");
		return "OK";
	}
}
```

- ObjectProvider를 통해 필요한 시점에 MyLogger를 요청함
- HTTP 요청이 들어온 뒤에 getObject()를 호출하므로 문제 없이 동작

```bash
[c8c52fc1-0ed3-4eb7-a002-cb2a12e28bc8] request scope bean create: hello.core.common.MyLogger@35777cb7
[c8c52fc1-0ed3-4eb7-a002-cb2a12e28bc8] [http://localhost:8080/log-demo] controller test
[c8c52fc1-0ed3-4eb7-a002-cb2a12e28bc8] [http://localhost:8080/log-demo] service id = testId
[c8c52fc1-0ed3-4eb7-a002-cb2a12e28bc8] request scope bean close: hello.core.common.MyLogger@35777cb7
```

- 즉, ObjectProvider 덕분에 ObjectProvider.getObject()를 호출하는 시점까지 request 스코프 빈을 스프링 컨테이너에 요청하는 것을 미룰 수 있음


#### 해결 방법 2. 프록시 객체 사용

```java
@Component
@Scope(value = "request", proxyMode = ScopedProxyMode.TARGET_CLASS)
public class MyLogger {...}
```

- proxyMode = ScopedProxyMode.TARGET_CLASS 옵션을 추가하면 스프링이 CGLIB 기반 프록시 객체를 만들어서 주입

- 프록시 객체는 진짜 MyLogger를 상속 받은  **`MyLogger$$SpringCGLIB$$0`** 라는 클래스
- 가짜 프록시 객체는 request 스코프와 관계 없고 위임 로직만 있음. 싱글톤처럼 동작함
- HTTP 요청과 무관하게 미리 주입이 가능하고, 실제 요청 시 진짜 객체를 찾아서 위임 처리
- 클라이언트 입장에서는 실제 객체와 차이가 없어 보임 (다형성)

**정리**

- ObjectProvider, 프록시 방식 모두 요청 시점까지 객체 생성을 지연할 수 있다는 공통점
- 프록시 방식은 어노테이션 한 줄로 해결 + DI 컨테이너와 다형성의 이점을 살릴 수 있다
