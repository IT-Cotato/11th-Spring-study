## 빈 생명주기 콜백의 필요성

애플리케이션 운영에서는 초기화/종료 작업이 필요함

- 애플리케이션 시작 시 DB 커넥션 풀 생성 
  - DB 커넥션을 생성할 때마다 TCP/IP 핸드쉐이킹이 발생함
  - 사용자 요청마다 연결 생성 시간이 추가되면 전체적인 성능이 떨어짐
  - 시작 사점에 미리 10~100개 정도의 커넥션을 생성해서 풀에 보관
  - 즉시 사용 가능한 커넥션을 제공해서 응답 시간을 단축할 수 있음

    
- 애플리케이션 종료 시 모든 연결의 안전한 종료
  - 사용 중인 리소스를 적절히 정리하지 않으면 메모리 누수 발생
  
  - 닫히지 않은 커넥션으로 인한 리소스 낭비를 방지

## 초기화 작업과 의존관계 주입 시점

생성자에서 초기화 작업을 수행하면 발생하는 문제

- 예시 코드(NetworkClient)에서 url 필드는 setter 메서드를 통해 설정됨
- 생성자 호출 시점에는 의존관계 주입이 완료되지 않은 상태

따라서 **의존관계 주입이 완료된 후**에 초기화 메서드가 호출되어야 함 !


> **스프링 빈의 생명주기**
>
> 스프링 컨테이너 생성 → 스프링 빈 생성 → 의존관계 주입 → **초기화 콜백** → 사용 → **소멸전 콜백** → 스프링 종료

> 객체의 생성과 초기화는 분리하는 것이 좋다.
초기화 작업이 단순하면 괜찮지만, 무거운 작업은 분리하여 최초 동작 전까지 지연시킬 수 있다.
>

## 빈 생명주기 콜백 지원 방법

### 1. 스프링 인터페이스 방식

```java
public class NetworkClient implements InitializingBean, DisposableBean {
   
    @Override
    public void afterPropertiesSet() throws Exception {
        // 초기화 작업
        connect();
        call("초기화 연결 메시지");
    }
    
    @Override
    public void destroy() throws Exception {
        // 종료 작업
        disconnect();
    }
}
```

- `InitializingBean`  → `afterPropertiesSet()` 메서드로 초기화를 지원한다.
- `DisposableBean` → `destroy()` 메서드로 소멸을 지원한다.
- 단점
    - 스프링 전용 인터페이스로 스프링에 코드가 의존하게 됨
    - 초기화, 소멸 메서드의 이름 변경 불가
    - 코드를 직접 고칠 수 없는 외부 라이브러리에는 적용할 수 없음

### 2. 설정 정보를 사용한 방식

```java
@Bean(initMethod = "init", destroyMethod = "close")
public NetworkClient networkClient() {
    NetworkClient networkClient = new NetworkClient();
    networkClient.setUrl("http://hello-spring.dev");
    return networkClient;
}
```
**장점**

- 메서드 이름을 자유롭게 가능
- 스프링 코드에 의존하지 않음
- 외부 라이브러리의 경우에도 초기화 메서드와 종료 메서드를 설정 정보에 적어주기만 하면 적용 가능

### 3. @PostConstruct, @PreDestroy 어노테이션 사용 (권장)

```java
import jakarta.annotation.PostConstruct;
import jakarta.annotatoin.PreDestroy;

public class NetworkClient {

    @PostConstruct
    public void init() {
        connect();
        call("초기화 연결 메시지");
    }
    
    @PreDestroy
    public void close() {
        disconnect();
    }
}
```

장점
- 자바 표준 기술로 스프링에 의존 X
- 컴포넌트 스캔과 잘 어울림
- 코드가 간결하고 직관적

단점
- 외부 라이브러리처럼 코드 수정이 불가한 경우엔 사용 불가 → 이땐 설정 정보를 사용한 방식으로!

> **컴포넌트 스캔과 잘 어울린다?**
> 
> 컴포넌트 스캔은 `@Component` 어노테이션을 통해 빈을 자동으로 등록하는 방식  
> 별도의 @Bean 설정 없이 빈을 등록함 
> 
>`@PostConstruct`와 `@PreDestroy`도 별도의 설정 클래스에서 initMethod나 destroyMethod 속성을 지정할 필요 없이 클래스 내부에서 바로 사용이 가능함
