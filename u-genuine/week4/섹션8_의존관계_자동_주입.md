## 의존관계 주입(DI) 방식

### 1. 생성자 주입

```java
@Component
public class OrderServiceImpl implements OrderService {
	private final MemberRepository memberRepository;
	private final DiscountPolicy discountPolicy;
	
	@Autowired
	public OrderServiceImpl(MemberRepository memberRepository, DiscountPolicy discountPolicy) {
			this.memberRepository = memberRepository;
			this.discountPolicy = discountPolicy;
		}
		// ...
}
```

- 객체 생성 시 생성자가 1회 호출되고, 스프링이 등록된 빈에서 파라미터에 맞는 객체를 찾아 주입해줌
- 생성자가 딱 1개인 경우 `@Autowired` 생략 가능
- 필수적이고 불변적인 의존 관계에 사용
- 스프링 빈 생명주기 - ‘스프링 빈 생성’ 단계에 실행됨
    - 객체 생성 시점에 모든 의존관계가 주입되므로 NPE 방지


> **스프링 빈 생명주기 (스프링 컨테이너의 빈 관리 과정)**
> 
> 스프링 컨테이너 생성 → **스프링 빈 생성** → **의존관계 주입** → 초기화 콜백 → 사용 → 소멸전 콜백 → 스프링 종료
> - 스프링 빈 생성 단계에서 **생성자 주입 실행**됨
> - 의존관계 주입 단계에서 **수정자 주입이 실행**됨


### 2. 수정자(Setter) 주입

```java
@Component
public class OrderServiceImpl implements OrderService {
	private MemberRepository memberRepository; // final 제거
	private DiscountPolicy discountPolicy; // final 제거
	
	@Autowired(required=false)
	public void setMemberRepository(MemberRepository memberRepository) {
			this.memberRepository = memberRepository;
	}
		
	@Autowired
	public void	setDiscountPolicy(DiscountPolicy discountPolicy) {
			this.discountPolicy = discountPolicy;
  }
		// ...
}
```

- 선택적이고 변경 가능한 의존관계에 사용
- @Autowired(required = false) : 해당 빈이 없어도 오류가 발생하지 않음.
- 스프링 빈 생명주기 - ‘의존관계 주입’ 단계에서 실행됨


> **스프링 컨테이너의 생성 과정**
>
> 1. 스프링 빈을 모두 컨테이너에 등록
>    - 생성자 주입은 이때 자동으로 발생 (객체 생성 시 생성자 호출)
> 2. 스프링 빈의 의존관계를 설정
>    - @Autowired가 붙어있는 setter 필드, 일반 메서드 주입은 이 단계에서 발생



### 3.  필드 주입

```java
@Component
public class OrderServiceImpl implements OrderService {
	@Autowired private MemberRepository memberRepository;
	@Autowired private DiscountPolicy discountPolicy;
	//...
}
```

- 필드에 직접 주입하는 방식
- 순수 자바 코드로 테스트 불가능
    - 필드가 private으로 선언되어 있어서 외부에서 직접 값을 설정할 수 없음
    - 스프링같은 DI 컨테이너 없이는 해당 필드에 의존성을 주입할 방법이 없음
    - 단위 테스트에서는 스프링 컨테이너를 실행하지 않고 테스트하는 것이 바람직한데, 필드 주입은 이를 어렵게 함 → 테스트 코드에서 저 필드를 사용하면 NPE 발생함
- 해결하기 위해서는 setter가 필요한데, 그럼 수정자 주입을 사용하는 것이 더 명확함

```java
// 테스트 코드
@Test
void fieldInjectionTest() {
		OrderServiceImpl orderService = new OrderServiceImpl(); 
			
		// 스프링 없이는 의존성을 주입할 방법이 없어 NPE 발생
		orderService.createOrder(1L, "itemA", 10000);
}
	
	
// OrderServiceImpl.java - createOrder() 메서드
public Order createOrder(Long memberId, String itemName, int itemPrice) {

		// memberRepository, discountPolicy를 사용하는데, 주입이 되지 않음
		Member member = memberRepository.findById(memberId);
		int discountPrice = discountPolicy.discount(member, itemPrice);
	
		return new Order(memberId, itemName, itemPrice, discountPrice);
}
```


### 4. 일반 메서드 주입

- 여러 필드를 한 번에 주입 가능
- 일반적으로 잘 사용하지 않는다

> 의존관계 주입은 스프링 컨테이너가 관리하는 스프링 빈이어야 동작함  
> 스프링 빈이 아닌 일반 객체에서는 (Member 등) @Autowired를 사용해도 아무 효과가 없음


### 생성자 주입 권장 이유
1. 불변성 보장

   final 키워드를 사용할 수 있어 객체의 불변성을 보장한다


2. 컴파일 시점 오류 확인 가능

   필수 의존성이 누락된 경우 컴파일 오류로 즉시 확인 가능

   수정자 주입은 런타임에 NPE등 예외로 확인됨

   

3. 테스트 용이성

   프레임워크에 의존하지 않고 순수 자바 코드로 단위 테스트 가능


### 롬복(Lombok) 활용

**lombok에서 제공해주는 어노테이션**

`@Getter` : 클래스에 붙이면 getXXX 메서드 자동 생성

`@Setter` : 클래스에 붙이면 setXXX 메서드 자동 생성

`@RequiredArgsConstructor` : 클래스에 붙이면 final 필드만 포함하는 생성자 자동 생성

- 생성자가 하나일 땐 @Autowired 생략 가능 → 생성자 코드 작성 없이 의존성 주입 가능

    ``` java
      @Component
      @RequiredArgsConstructor
      public class OrderServiceImpl implements OrderService {
      private final MemberRepository memberRepository;
      private final DiscountPolicy discountPolicy;
      // 생성자 코드 없이도 의존성 주입이 이루어짐
      }
    ```
- 생성자 코드 없이 의존성 주입 이루어짐

## 빈 중복 문제

@Autowired은 타입으로 주입할 객체를 조회하는데, 같은 타입의 빈이 여러개 등록되면
`NoUniqueBeanDefinitionExcepiotn` 오류가 발생함

```java
@Component
public class FixDiscontPolicy implement DiscountPolicy {}

@Component
public class RateDiscontPolicy implement DiscountPolicy {}
```


### 해결 방법 1. @Autowired 필드명을 통한 매칭

같은 타입의 빈(자식 포함)이 여러개 등록된 경우 **필드명**이나 **파라미터 이름**을 매핑한다

```java
@Component
public class OrderServiceImpl implements OrderService {
	private final MemberRepository memberRepository;
	private final DiscountPolicy discountPolicy;

	@Autowired
	public OrderServiceImpl(
		MemberRepository memberRepository,
		DiscountPolicy rateDiscountPolicy // RateDiscountPolicy
	) {
		this.memberRepository = memberRepository;
		this.discountPolicy = rateDiscountPolicy;
	}

}
```


### 해결 방법 2. @Qualifier 사용

빈에 추가 구분자를 붙여 주입하는 방법

```java
// 빈 등록 시
@Component
@Qualifier("fixDiscountPolicy")
public class FixDiscontPolicy implement DiscountPolicy {}

@Component
@Qualifier("mainDiscountPolicy")
public class RateDiscontPolicy implement DiscountPolicy {}

// 주입 시
@Autowired
	public OrderServiceImpl(
		MemberRepository memberRepository, 
		@Qualifier("mainDiscountPolicy") DiscountPolicy discountPolicy // RateDiscountPolicy
		) {
			this.memberRepository = memberRepository;
			this.discountPolicy = discountPolicy;
		}
```

- 주입 받는 모든 코드에 @Qualifier 를 붙여야 한다는 단점이 있음


### 해결 방법 3. @Primary 사용

여러 빈 중 우선순위를 지정하는 방법

```java
@Component
public class FixDiscontPolicy implement DiscountPolicy {}

@Component
@Primary
public class RateDiscontPolicy implement DiscountPolicy {} 
```

- `@Primary` 가 붙은 빈이 우선적으로 주입 됨
- `@Qualifier`가 `@Primary`보다 우선순위가 높음 (수동 > 자동)


### 해결 방법 4. 커스텀 어노테이션 생성

`@Qualifier(”aaaaa”)` - 문자열에 오타가 있을 때 컴파일 시점에 확인할 수 없다는 단점

```java
// 기존 @Qualifer 어노테이션
@Target({ElementType.FIELD, ElementType.METHOD, ElementType.PARAMETER, ElementType.TYPE, ElementType.ANNOTATION_TYPE})
@Retention(RetentionPolicy.RUNTIME)
@Inherited
@Documented
public @interface Qualifier { 
    String value() default "";
}

// 커스텀 어노테이션 정의
@Target({ElementType.FIELD, ElementType.METHOD, ElementType.PARAMETER, ElementType.TYPE, ElementType.ANNOTATION_TYPE})
@Retention(RetentionPolicy.RUNTIME)
@Inherited
@Documented
@Qualifier("mainDiscountPolicy")
public @interface MainDiscountPolicy {
}

// 사용
@Component
@MainDiscountPolicy
public class RateDiscountPolicy implements DiscountPolicy {}

// 주입
@Autowired
public OrderServiceImpl(
    MemberRepository memberRepository, 
    @MainDiscountPolicy DiscountPolicy discountPolicy
) {
    this.memberRepository = memberRepository;
    this.discountPolicy = discountPolicy;
}
```

- 기존 @Qualifier 어노테이션과 동일한 동작을 위해 어노테이션을 그대로 가져옴
- `@MainDiscountPolicy`는 `@Qualifier(”mainDiscountPolicy)`어노테이션과 동일함
- 커스텀 어노테이션을 사용하면 코드의 가독성이 높아지고, 오타로 인한 오류를 컴파일 시점에 방지할 수 있게 되었다.
