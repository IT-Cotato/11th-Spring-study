# 섹션 7. 컴포넌트 스캔

### 컴포넌트 스캔
스프링은   
설정 정보 없이도 자동으로 스프링 빈을 등록하는 컴포넌트 스캔 기능과
의존관계를 자동으로 주입하는 @Autowired 기능을 제공함

```java
@Configuration
@ComponentScan(
    excludeFilters = @Filter(type = FilterType.ANNOTATION, classes = Configuration.class)
)
public class AutoAppConfig {
    // 빈 등록 코드가 없음
}
```
- @ComponentScan 어노테이션만 붙이면 컴포넌트 스캔 기능이 활성화됨
- 별도의 @Bean 등록 코드가 필요 없음


### 컴포넌트 스캔 대상
@Component 어노테이션이 붙은 클래스를 스캔해서 스프링 빈으로 등록함
```java
@Component
public class MemoryMemberRepository implements MemberRepository {}
```
```java
@Component
public class RateDiscountPolicy implements DiscountPolicy {}
```
```java
@Component
public class MemberServiceImpl implements MemberService {
    private final MemberRepository memberRepository;
    
    @Autowired  // 자동 의존관계 주입
    public MemberServiceImpl(MemberRepository memberRepository) {
        this.memberRepository = memberRepository;
    }
}
```

### 의존관계 자동 주입
스프링에서는 의존관계를 주입하는 방식이 두 가지로 나뉜다.  

**1. 수동 의존관계 주입 (설정 정보 사용)**  

기존 방식에서는 @Configuration 클래스 내부의 @Bean 메서드에서 명시적으로 의존 관계를 주입했음
```java
@Configuration
public class AppConfig {
    @Bean
    public MemberService memberService() {
        // 명시적으로 의존관계를 주입
        return new MemberServiceImpl(memberRepository());
    }
    
    @Bean
    public MemberRepository memberRepository() {
        return new MemoryMemberRepository();
    }
}
```
이 방식에서는 설정 클래스인 AppConfig가 명확하게 어떤 구현체를 주입할지 코드로 명시하고 있음

**2. 자동 의존관계 주입 (@Autowired)**

컴포넌트 스캔 방식에서는 명시적인 의존관계 주입 코드가 없음
```java
@Configuration
@ComponentScan
public class AutoAppConfig {
    // 빈 등록 및 의존관계 주입 코드가 없음
}
```
이제 각 컴포넌트는 스스로 필요한 의존관계를 주입받아야 하는데, 이때 @Autowired 사용

```java
@Component
public class MemberServiceImpl implements MemberService {
    private final MemberRepository memberRepository;
    
    @Autowired  // 스프링이 자동으로 의존관계를 주입
    public MemberServiceImpl(MemberRepository memberRepository) {
        this.memberRepository = memberRepository;
    }
}
```

스프링은 같은 타입의 빈을 찾아서 주입
- MemberRepository 타입을 주입할 때 MemoryMemberRepository가 주입됨
- 같은 타입의 빈이 여러 개 있으면 충돌 발생

### 컴포넌트 스캔 탐색 위치
- basePackages, basePackageClasses 옵션으로 탐색 시작 위치를 지정할 수 있음
- 지정하지 않으면 @ComponentScan이 붙은 클래스의 패키지가 시작 위치
- 권장 방법: 설정 정보 클래스를 프로젝트 최상단에 배치 (관례)
  스프링 부트는 @SpringBootApplication에 @ComponentScan이 포함되어 있어, 이 어노테이션을 프로젝트 루트에 위치시키는 것이 관례

### 중복 등록과 충돌
컴포넌트 스캔에서 같은 빈 이름을 등록할 경우  
**1. 자동 빈 등록 vs 자동 빈 등록**
   여러 클래스에 같은 빈 이름을 부여할 경우(예: @Component("service"))    
   ConflictingBeanDefinitionException 예외가 발생

**2. 수동 빈 등록 vs 자동 빈 등록**
```java
@Component  // memoryMemberRepository 이름으로 자동 등록
public class MemoryMemberRepository implements MemberRepository {}

// 설정 클래스에서 수동 등록
@Bean(name = "memoryMemberRepository") 
public MemberRepository memberRepository() {
    return new MemoryMemberRepository();
}
```
- 기존에는 수동 빈 등록이 우선권을 가졌음(자동 빈을 수동 빈이 오버라이딩)
- 하지만 현실에서 의도치 않은 오버라이딩으로 오류 발생이 잦아서
- 스프링 부트에서는 기본적으로 오류가 발생하도록 설정되어 있음
