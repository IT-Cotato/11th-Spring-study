### 💡 **빈 스코프**

- 스프링 빈은 기본적으로 싱글톤 스코프로 생성됨
- 스코프 : 빈이 존재할 수 있는 범위
- 다양한 스코프 존재
    - 싱글톤 : 기본 스코프, 스프링 컨테이너의 시작과 종료까지 유지되는 가장 넓은 범위
    - 프로토타입 : 스프링 컨테이너는 프로토타입 빈의 생성과 의존관계 주입까지만 관여
    - 웹 관련 스코프
        - request : 웹 요청이 들어오고 나갈 때까지만 유지됨
        - session, application

### 💡  프로토타입 스코프

- 싱글톤 스코프의 빈을 조회하면 스프링 컨테이너는 항상 같은 인스턴스의 스프링 빈을 반환 BUT, 프로토타입 스 코프를 스프링 컨테이너에 조회하면 스프링 컨테이너는 항상 새로운 인스턴스를 생성해서 반환해줌
- 싱글톤 빈 요청
    - 스프링 컨테이너는 본인이 관리하는 스프링 빈을 반환함
    - 스프링 컨테이너에 같은 요청이 와도 같은 객체 인스턴스의 스프링 빈을 반환
    - 종료 메서드까지 정상 호출됨

    ```java
    SingletonBean.init
    singletonBean1 = hello.core.scope.SingletonTest$SingletonBean@3a12c404
    singletonBean2 = hello.core.scope.SingletonTest$SingletonBean@3a12c404
    SingletonBean.destroy
    ```

- 프로토타입 빈 요청
    - 스프링 컨테이너는 프로토타입 스코프의 빈을 스프링 컨테이너에 요청하는 시점에 프로토타입 빈을 생성하고, 필요한 의존관계를 주입
    - 프로토타입 스코프의 빈은 스프링 컨테 이너에서 빈을 조회할 때 생성되고, 초기화 메서드도 실행됨
    - 스프링 컨테이너에 같은 요청이 오면 항상 새로운 프로토타입 빈을 생성해서 반환
    - 스프링 컨테이너가 종료될 때 @PreDestroy 같은 종료 메서드가 실행되지 않음

    ```java
    find prototypeBean1
    SingletonBean.init
    find prototypeBean2
    SingletonBean.init
    prototypeBean1 = hello.core.scope.PrototypeTest$PrototypeBean@3a12c404
    prototypeBean2 = hello.core.scope.PrototypeTest$PrototypeBean@1941a8ff
    ```

- 프로토타입 빈
    - 스프링 컨테이너는 프로토타입 빈을 생성하고, 의존관계 주입, 초기화까지만 처리함
    - 스프링 컨테이너는 생성된 프로토타입 빈을 관리하지 않음
    - 프로토타입 빈은 프로토타입 빈을 조회한 클라이언트가 관리해야 하므로 종료 메서드에 대한 호출도 클라이 언트가 직접 해야함

### 💡 **프로토타입 스코프** - **싱글톤 빈과 함께 사용시 문제점**

- 프로토타입 스코프의 빈을 싱글톤 빈과 함께 사용할 때는 의도한 대로 잘 동작하지 않으므로 주의해야함
- clientBean은 싱글톤이므로, 보통 스프링 컨테이너 생성 시점에 함께 생성되고, 의존관계 주입도 발생
- clientBean은 프로토타입 빈을 내부 필드에 보관함
- clientBean이 내부에 가지고 있는 프로토타입 빈은 이미 과거에 주입이 끝난 빈임
- 주입 시점에 스프링 컨테이너에 요청해서 프로토타입 빈이 새로 생성됨
- 우리는 프로토타입 빈을 주입 시점에만 새로 생성하는 것이 아닌, 사용할 때마다 새로 생성하고 사용하는 것을 원함

### 💡 **프로토타입 스코프** - **싱글톤 빈과 함께 사용시** Provider**로 문제 해결**

- 가장 간단한 해결 방법은 싱글톤 빈이 프로토타입을 사용할 때 마다 스프링 컨테이너에 새로 요청하는 것
    - ac.getBean()을 통해서 항상 새로운 프로토타입 빈이 생성됨
    - 직접 필요한 의존관계를 찾는 것을 Dependency Lookup(DL) 의존관계 조회(탐색)이라고 함
    - 단점 ) 스프링 컨테이너에 종속적인 코드가 됨

    ```java
    @Autowired
    private ApplicationContext ac;
    
    public int logic() {
       PrototypeBean prototypeBean = ac.getBean(PrototypeBean.class);
       prototypeBean.addCount();
       int count = prototypeBean.getCount();
       return count;
    }
    ```

- ObjectProvider(ObjectFactory)
    - 지정한 빈을 컨테이너에서 대신 찾아주는 DL 서비스를 제공해줌
    - prototypeBeanProvider.getObject() 을 통해서 항상 새로운 프로토타입 빈이 생성됨
    - ObjectProvider의 getObject() 를 호출하면 내부에서는 스프링 컨테이너를 통해 해당 빈을 찾아서 반환gowna ⇒ DL

    ```java
    @Autowired
    private Provider<PrototypeBean> prototypeBeanProvider;
    
    public int logic() {
        PrototypeBean prototypeBean = prototypeBeanProvider.get();
        prototypeBean.addCount();
        int count = prototypeBean.getCount();
        return count;
    }
    ```

- JSR-330 Provider
    - 자바 표준을 사용하는 방법
    - gradle 에 추가

        ```java
        implementation 'jakarta.inject:jakarta.inject-api:2.0.1'
        ```

    - provider.get() 을 통해서 항상 새로운 프로토타입 빈이 생성됨
    - provider의 get() 을 호출하면 내부에서는 스프링 컨테이너를 통해 해당 빈을 찾아서 반환해줌 ⇒ DL
    - 자바 표준이므로 스프링이 아닌 다른 컨테이너에서도 사용 가능

    ```java
     @Autowired
     private Provider<PrototypeBean> provider;
     public int logic() {
         PrototypeBean prototypeBean = provider.get();
         prototypeBean.addCount();
         int count = prototypeBean.getCount();
         return count;
    }
    ```

### 💡 **웹 스코프**

- 웹 환경에서만 동작함
- 스프링이 해당 스코프의 종료시점까지 관리 ⇒ 종료 메서드 호출됨
- 웹 스코프 종류 중 request 는 HTTP 요청 하나가 들어오고 나갈 때 까지 유지되는 스코프, 각각의 HTTP 요청마다 별도의 빈 인스턴스가 생성되고, 관리됨
- gradle 에 추가 : 웹 환경에서만 동작하므로 라이브러리 설치

    ```java
    implementation 'org.springframework.boot:spring-boot-starter-web'
    ```

- HTTP 요청 로그 출력
    - MyLogger, LogDemoController, LogDemoService 사용
    - request 스코프 빈은 아직 생성되지 않음, 이 빈은 실제 고객의 요청이 와야 생성됨 ⇒ 애플리케이션 실행 시점에 오류 발생
    - Provider를 사용해 해결
        - ObjectProvider.getObject() 를 호출하는 시점까지 request scope 빈의 생성을 지연
    - 프록시를 사용해 해결
        - @Scope(value = "request", proxyMode = ScopedProxyMode.TARGET_CLASS)
        - MyLogger의 가짜 프록시 클래스를 만들어두고 HTTP request와 상관 없이 가짜 프록시 클래스를 다른 빈에 미리 주입해 둘 수 있음
        - @Scope 의 proxyMode = ScopedProxyMode.TARGET_CLASS) 를 설정하면 스프링 컨테이너는 CGLIB(바이트코드를 조작하는 라이브러리)를 사용해서, MyLogger를 상속받은 가짜 프록시 객체를 생성
        - 가짜 프록시 객체는 요청이 오면 그때 내부에서 진짜 빈을 요청하는 위임 로직이 있음

        ```java
        myLogger = class hello.core.common.MyLogger$$SpringCGLIB$$0
        [8057966c-79d3-47b8-a58c-fc3ddae6a311] request scope bean create:hello.core.common.MyLogger@1c55080e
        [8057966c-79d3-47b8-a58c-fc3ddae6a311]http://localhost:8080/log-demo]controller test
        [8057966c-79d3-47b8-a58c-fc3ddae6a311]http://localhost:8080/log-demo]service id = testId
        [8057966c-79d3-47b8-a58c-fc3ddae6a311] request scope bean create:hello.core.common.MyLogger@1c55080e
        ```

    - 코드 (Proxy version)
        - MyLogger

            ```java
            @Component
            @Scope(value = "request", proxyMode = ScopedProxyMode.TARGET_CLASS)
            public class MyLogger {
            
                private String uuid;
                private String requestURL;
            
                public void setRequestURL(String requestURL) {
                    this.requestURL = requestURL;
                }
            
                public void log(String message) {
                    System.out.println("[" + uuid + "]" + requestURL + "]" + message);
                }
            
                @PostConstruct
                public void init() {
                    uuid = UUID.randomUUID().toString();
                    System.out.println("[" + uuid + "] request scope bean create:" + this);
                }
            
                @PreDestroy
                public void close() {
                    System.out.println("[" + uuid + "] request scope bean create:" + this);
                }
            }
            
            ```

        - LogDemoService

            ```java
            @Service
            @RequiredArgsConstructor
            public class LogDemoService {
                private final MyLogger myLogger;
                public void logic(String id) {
                    myLogger.log("service id = " + id);
                }
            }
            ```

        - LogDemoController

            ```java
            @Controller
            @RequiredArgsConstructor
            public class LogDemoController {
                private final LogDemoService logDemoService;
                private final MyLogger myLogger;
            
                @RequestMapping("log-demo")
                @ResponseBody
                public String logDemo(HttpServletRequest request) throws InterruptedException {
                    String requestURL = request.getRequestURL().toString();
            
                    // 껍데기 myLogger
                    // myLogger = class hello.core.common.MyLogger$$SpringCGLIB$$0
                    System.out.println("myLogger = " + myLogger.getClass());
                    myLogger.setRequestURL(requestURL);
            
                    myLogger.log("controller test");
                    Thread.sleep(100);
                    logDemoService.logic("testId");
                    return "OK";
                }
            }
            
            ```

### 💡 새롭게 알게 된 점 및 중요한 점 정리

1. 빈 스코프
    - 빈이 존재할 수 있는 범위
    - 종류 : 싱글톤, 프로토타입, 웹 관련 스코프(request 등)
2. 프로토타입 빈
    - 프로토타입 스코프를 스프링 컨테이너에 조회하면 스프링 컨테이너는 항상 새로운 인스턴스를 생성해서 반환
    - 스프링 컨테이너는 프로토타입 빈을 생성하고, 의존관계 주입, 초기화까지만 처리함
3. 프로토타입 스코프과 싱글톤 빈과 함께 사용시
    - 문제점 ) 주입 시점에 스프링 컨테이너에 요청해서 프로토타입 빈이 새로 생성이 된 것이지, 사용 할 때마다 새로 생성되는 것이 아님
    - 해결 ) 스프링 컨테이너에 매번 요청, ObjectProvider(ObjectFactory), Provider
        - ObjectProvider : 지정한 빈을 컨테이너에서 대신 찾아주는 DL 서비스를 제공해줌, getObject()을 통해서 항상 새로운 프로토타입 빈이 생성됨
        - Provider : 자바 표준, get() 을 호출하면 내부에서는 스프링 컨테이너를 통해 해당 빈을 찾아서 반환해줌(DL), get()을 통해서 항상 새로운 프로토타입 빈이 생성
4. HTTP 요청 로그 출력(웹 스코프 예제)
    - 문제점 ) 스프링 애플리케이션을 실행하는 시점에 request 스코프 빈은 아직 생성되지 않아서 오류 발생
    - 해결 ) ObjectProvider, Proxy
        - Proxy : @Scope(value = "request", proxyMode = ScopedProxyMode.TARGET_CLASS), HTTP request와 상관 없이 가짜 프록시 클래스를 다른 빈에 미리 주입해둘 수 있음
