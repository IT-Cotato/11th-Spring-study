### 💡 JPA에서 중요한 2가지

- 객체와 관계형 데이터베이스 매핑하기(Object Relational Mapping)
- 영속성 컨텍스트
    - 엔티티를 영구 저장하는 환경
    - EntityManager.persist(entity); → db에 저장하는게 아니라 영속성 컨텍스트에 저장하는 것
    - 엔티티 매니저를 통해서 영속성 컨텍스트에 접근
    - 엔티티 매니저를 만들면 영속성 컨텍스트가 생김

---

### 💡 엔티티의 생명주기

- 비영속 (new/transient) : 영속성 컨텍스트와 관계가 없는 상태

    ```jsx
    // 비영속 상태
    Member member = new Member();
    member.setId(101L);
    member.setName("HelloJPA");
    ```

- 영속 (managed) : 영속성 컨텍스트에 관리되는 상태
    - 엔티티 매니저에 persist하면 엔티티 매니저 안에 영속
    - 컨텍스트 안에 객체가 들어가면서 영속 상태가 됨
    - DB에 들어가는 건 commit 할 때

    ```jsx
    // 1차 캐시에 저장됨
    em.persist(member);
    ```

- 준영속 (detached) : 영속성 컨텍스트에 저장되었다가 분리된 상태
    - 영속성 컨텍스트가 제공하는 기능 사용 못 함

    ```jsx
    em.detach(member); // 특정 엔티티만 준영속 상태로 전환
    em.clear(); // 영속성 컨텍스트 초기화
    em.close(); // 영속성 컨텍스트 종료
    ```

- 삭제 (removed) : 삭제된 상태

    ```jsx
    em.remove(member);
    ```

### 💡 영속성 컨텍스트의 이점

- 애플리케이션이랑 DB 사이에 있는 것의 이점이 있음
- 1차 캐시
    - 애플리케이션에서 전체 공유하는게 아닌 한 트랜잭션 안에서만 효과가 있으므로 성능 이점을 크게 얻을 수는 없음 (복잡할 때는 도움이 될 수 있음)
    - DB에서 찾기 전에 1차 캐시에 존재하는지 먼저 찾음, 1차 캐시에 없으면 DB에서 조회하고 1차 캐시에 저장하고 반환해줌
- 동일성 보장
    - 같은 트랜잭션 내에서는 동일한 엔티티를 조회할 때 항상 동일한 객체를 반환하여, 엔티티의 동일성이 유지됨
- 트랜잭션을 지원하는 쓰기 지연
    - 엔티티 매니저는 데이터 변경시 트랜잭션을 시작해야함
    - persist(memberA)하면 1차 캐시에 저장하고 INSERT SQL 생성해서 쓰기 지연 SQL 저장소에 넣어놓음
    - 커밋하는 순간 데이터베이스에 INSERT SQL을 보냄

    ```jsx
    EntityTransaction tx = em.getTransaction();Add commentMore actions
    tx.begin();
    em.persist(memberA);
    em.persist(memberB);
    tx.commit();
    ```

- 변경 감지(Dirty Checking)
    - 영속성 컨텍스트가 관리하는 엔티티의 내용이 변경되었을 때, 개발자가 별도의 update 호출 없이도 데이터베이스에 자동으로 반영되는 기능
    - 값을 바꾸면 트랜잭션이 커밋되는 시점에 변경을 반영하므로 em.persist()을 따로 안 해야함
    - 1차 캐시에 들어온 상태를 스냅샷으로 1차 캐시에서 보관하는데 JPA가 엔티티와 스냅샷을 비교해서 변경된 부분이 있으면 이를 감지해 update 쿼리를 데이터베이스에 반영하고 커밋
- 지연 로딩(Lazy Loading)

---

### 💡 플러시

- 영속성 컨텍스트의 변경내용을 데이터베이스에 반영
- 데이터베이스가 커밋될 때 자동으로 플러시가 발생하면서 영속성 컨텍스트의 쿼리를 DB에 날려줌
- 쓰기 지연 SQL 저장소의 쿼리를 데이터베이스에 전송 (등록, 수정, 삭제 쿼리)
- 영속성 컨텍스트를 플러시하는 방법
    - 직접 호출 : em.flush()
    - 트랜잭션 커밋 : 플러시 자동 호출
    - JPQL 쿼리 실행 : 플러시 자동 호출
- 플러시 모드 옵션 : 기본 값인 FlushModeType.AUTO 쓰는 것을 권장
- 영속성 컨텍스트를 비우지 않음
- 영속성 컨텍스트의 변경 내용을 데이터베이스에 동기화

### 💡 새롭게 알게 된 점 및 중요한 점 정리

1. 영속성 컨텍스트 : 엔티티를 영구 저장하는 환경으로, 엔티티 매니저를 통해서 영속성 컨텍스트에 접근할 수 있음
2. 엔티티 매니저는 데이터 변경시 트랜잭션을 시작해야함
3. 1차 캐시에 들어온 상태를 스냅샷으로 1차 캐시에서 보관하는데 JPA가 엔티티와 스냅샷을 비교해서 변경된 부분이 있으면 이를 감지해 update 쿼리를 데이터베이스에 반영하고 커밋
4. 플러시 : 영속성 컨텍스트의 변경내용을 데이터베이스에 반영, 쓰기 지연 SQL 저장소의 쿼리를 데이터베이스에 전송 (등록, 수정, 삭제 쿼리), 커밋 전에 자동으로 호출됨
