### 💡 웹 애플리케이션과 싱글톤

- 웹 애플리케이션은 보통 여러 고객이 동시에 요청을 함
- 만약 memberService가 3번 요청되면 3개의 memberService 객체가 만들어짐
- 호출을 할 때마다 객체를 새로 생성함

    ```java
    MemberService memberService1 = appConfig.memberService();
    MemberService memberService2 = appConfig.memberService();
    System.out.println("memberService1 = " + memberService1);
    System.out.println("memberService2 = " + memberService2);
    ->
    참조값이 다르다는 걸 확인할 수 있음
    ```

- 이 상황을 해결하려면 사용하는 해당 객체를 딱 1개만 생성하고 필요할 때마다 이 객체를 공유하도록 설계하면 됨 → 싱글톤!

### 💡  싱글톤

- 싱글톤 패턴
    - 클래스의 인스턴스가 딱 1개만 생성되는 것을 보장하는 디자인 패턴
    - 객체 인스턴스를 2개 이상 생성하지 못하도록 막아야 하므로 private 생성자를 사용해서 외부에서 new 키워드를 사용해서 생성자에 접근하는 것을 막음

        ```java
        public class SingletonService {
        
        	private static final SingletonService instance = new SingletonService();
        	
        	public static SingletonService getInstance() {
        	    return instance;
        	}
        	
        	private SingletonService() {
        	}
        	public void logic() { 
        		System.out.println("싱글톤 객체 로직 호출");
        	} 
        }
        ```

        ```java
        // getInstance()를 했을 때 만들어져 있지 않으면 그때 만들고
        // 만들어져 있으면 있는 것을 가져옴
        SingletonService singletonService1 = SingletonService.getInstance();
        SingletonService singletonService2 = SingletonService.getInstance();
        
        System.out.println("singletonService1 = " + singletonService1);
        System.out.println("singletonService2 = " + singletonService2);
        assertThat(singletonService1).isSameAs(singletonService2);
        ->
        호출할 때마다 같은 객체 인스턴스를 반환한다는 것을 알 수 있음
        ```

- 싱글톤 패턴 문제점
    - 의존관계상 클라이언트가 구체 클래스에 의존하기 때문에 DIP를 위반함
    - 클라이언트가 구체 클래스에 의존해서 OCP 원칙을 위반할 가능성이 높음
    - private 생성자로 자식 클래스를 만들기 어려움
    - 이런 많은 문제점이 있어서 안티패턴으로 불리기도 함
- 그러나!! 스프링 컨테이너는 싱글톤 패턴 문제점을 다 해결하면서 객체를 다 싱글톤으로 만들어서 관리해줌

### 💡 싱글톤 컨테이너

- 스프링 컨테이너는 내가 따로 싱글턴 패턴을 적용하지 않아도 객체 인스턴스를 싱글톤으로 관리함
- 지금까지 우리가 봤던 스프링 빈이 바로 싱글톤으로 관리되는 빈
- 스프링 컨테이너는 싱글톤 컨테이너 역할을 함
- 스프링 컨테이너 덕분에 요청이 올 때마다 객체를 생성하지 않고 이미 만들어진 객체를 공유해서 사용하므로 요청에 빠르게 대응할 수 있고 효율적으로 재사용할 수 있음

```java
ApplicationContext ac = new AnnotationConfigApplicationContext(AppConfig.class);
MemberService memberService1 = ac.getBean("memberService", MemberService.class);
MemberService memberService2 = ac.getBean("memberService", MemberService.class);
System.out.println("memberService1 = " + memberService1);
System.out.println("memberService2 = " + memberService2);
assertThat(memberService1).isSameAs(memberService2);
->
참조값이 같으므로 같은 인스턴스를 반환한다는 것을 알 수 있음
```

참고) 스프링의 기본 빈 등록 방식은 싱글톤임(99%) 싱글톤 방식 말고 요청할 때마다 새로운 객체를 생성해서 반환하는 기능도 제공하는데 이건 빈 스코프에서 설명

### 💡 싱글톤 방식의 주의점

- 무상태로 설계해야함
    - 여러 클라이언트가 하나의 객체 인스턴스를 공유하므로 싱글톤 객체는 상태를 유지(stateful)하게 설계하면 안됨
    - 특정 클라이언트에 의존적인 필드가 있으면 안됨
    - 특정 클라이언트가 값을 변경할 수 있는 필드가 있으면 안됨

    ```java
    public class StatefulService {
    
    	private int price;
    	
    	public void order(String name, int price) { 
    		System.out.println("name = " + name + " price = " + price);
    		this.price = price;
    	}
         public int getPrice() {
             return price;
    	} 
    }
    ```

    ```java
    public class StatefulServiceTest {
        @Test
        void statefulServiceSingleton() {
            AnnotationConfigApplicationContext ac = new AnnotationConfigApplicationContext(TestConfig.class);
            StatefulService statefulService1 = ac.getBean(StatefulService.class);
            StatefulService statefulService2 = ac.getBean(StatefulService.class);
    
            //ThreadA : A사용자 10000원 주문
            statefulService1.order("userA", 10000);
            //ThreadB : B사용자 20000원 주문
            statefulService2.order("userB", 20000);
    
            //ThreadA : A사용자 주문 금액 조회
            int price = statefulService1.getPrice();
            System.out.println("price = " + price);
    
    				// statefulService1 이므로 10000을 기대했는데 20000임
            Assertions.assertThat(statefulService1.getPrice()).isEqualTo(20000);
    
        }
        static class TestConfig {
            @Bean
            public StatefulService statefulService() {
                return new StatefulService();
            }
        }
    }
    ```

### 💡 @Configuration 과 싱글톤

- 상황 : 다른 서비스 두개에서 같은 MemoryMemberRepository 를 생성함
- 결과 : memberRepository 인스턴스는 모두 같은 인스턴스가 공유되어 사용
- 스프링 컨테이너가 싱글톤을 보장해줌

    ```java
    @Test
    void configurationTest() {
        AnnotationConfigApplicationContext ac = new AnnotationConfigApplicationContext(AppConfig.class);
        MemberServiceImpl memberService = ac.getBean("memberService", MemberServiceImpl.class);
        OrderServiceImpl orderService = ac.getBean("orderService", OrderServiceImpl.class);
        MemberRepository memberRepository = ac.getBean("memberRepository", MemberRepository.class);
        MemberRepository memberRepository1 = memberService.getMemberRepository();
        MemberRepository memberRepository2 = orderService.getMemberRepository();
        System.out.println("memberService -> memberRepository = " + memberRepository1);
        System.out.println("orderService -> memberRepository = " + memberRepository2);
        System.out.println("memberRepository = " + memberRepository);
    
        assertThat(memberService.getMemberRepository()).isSameAs(memberRepository);
        assertThat(orderService.getMemberRepository()).isSameAs(memberRepository);
    }
    ->
    같은 객체 인스턴스를 반환한다는 것을 알 수 있음
    ->
    call AppConfig.memberService
    call AppConfig.memberRepository
    call AppConfig.orderService
    ->
    실행결과를 통해 스프링 컨테이너가 싱글톤 보장해준다는 것을 알 수 있음
    ```

### 💡  @Configuration과 바이트코드 조작의 마법

- 스프링은 스프링 빈이 싱글톤이 되도록 보장해주기 위해서 클래스의 바이트코드를 조작하는 CGLIB 라이브러리를 사용함
- 바이트 조작 라이브러리를 사용해서 AppConfig 클래스를 상속받은 임의의 다른 클래스(AppConfig$$SpringCGLIB$$0)를 만들고 그 다른 클래스를 스프링 빈으로 등록 → 즉, AppConfig를 스프링 빈으로 등록한 것이 아닌 AppConfig$$SpringCGLIB$$0를 스프링 빈으로 등록한 것

    ```java
    @Test
    void configurationDeep() {
        AnnotationConfigApplicationContext ac = new AnnotationConfigApplicationContext(AppConfig.class);
        AppConfig bean = ac.getBean(AppConfig.class);
        // bean = class hello.core.AppConfig$$SpringCGLIB$$0 출력됨
        // 내가 만든 클래스가 아니라 스프링이 AppConfig 클래스를 상속받은 임의의 다른 클래스를 만들고
        // 그 다른 클래스를 스프링 빈으로 등록한 것 -> 이 클래스가 싱클톤이 보장되도록 해줌
        System.out.println("bean = " + bean.getClass());
    }
    ```

- AppConfig@CGLIB 예상 코드
    - @Bean이 붙은 메서드마다 이미 스프링 빈이 존재하면 존재하는 빈을 반화하고, 스프링 빈이 없으면 생성해서 스프링 빈으로 등록하고 반환 → 싱글톤 보장
- AppConfig에 @Configuration를 붙이지 않으면?
    - @Configuration 을 붙이면 바이트코드를 조작하는 CGLIB 기술을 사용해서 싱글톤을 보장
    - 근데 @Configuration을 붙이지 않고 @Bean만 적용하면 아까와 달리 ‘call AppConfig.memberRepository’ 가 3번 호출됨
    - 인스턴스도 각각 만들어짐
    - @Bean만 사용해도 스프링 빈으로 등록되지만, 싱글톤은 보장하지 않음
    - 따라서 싱글톤을 보장하기 위해서 스프링 설정 정보는 항상 @Configuration을 붙여야 함

### 💡 새롭게 알게 된 점 및 중요한 점 정리

1. 스프링 컨테이너는 객체 인스턴스를 싱글톤으로 관리하므로 요청이 올 때마다 객체를 생성하지 않고 이미 만들어진 객체를 공유해서 사용함
2. 싱글톤 방식은 여러 클라이언트가 하나의 객체를 공유해서 사용하므로 싱글톤 객체가 무상태를 유지하도록 해야함
3. 스프링 컨테이너는 CGLIB 라이브러리를 사용해서 AppConfig 클래스를 상속받은 임의의 클래스를 스프링 빈으로 등록하면서 싱글톤이 가능하게 함
4. @Configuration 을 AppConfig에 붙이면 바이트 코드를 조작하는 CGLIB 기술을 사용해서 싱글톤을 보장함 따라서 싱글톤을 보장하기 위해서는 스프링 설정 정보에 항상 @Configuration을 붙여야 함
