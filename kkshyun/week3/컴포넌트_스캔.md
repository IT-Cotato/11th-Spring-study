### 💡 컴포넌트 스캔과 의존관계 자동 주입 시작하기

- 지금까지는 직접 스프링 빈을 등록하였는데 스프링 빈의 개수가 많아지면 일일이 등록하기도 귀찮고 설정 정보도 커지고, 누락하는 문제도 발생할 수 있음
- 스프링은 설정 정보가 없어도 자동으로 자동으로 스프링 빈을 등록하는 컴포넌트 스캔이라는 기능을 제공함
- 의존관계도 자동으로 주입하는 @Autowired라는 기능도 제공함
- 컴포넌트 스캔
    - 컴포넌트 스캔을 사용하려면 먼저 @ComponentScan을 설정 정보에 붙여주면 됨
    - 컴포넌트 스캔은 이름 그대로 @Component 애노테이션이 붙은 클래스를 스캔해서 스프링 빈으로 등록
    - @ComponentScan 은 @Component가 붙은 모든 클래스를 스프링 빈으로 등록함
        - 클래스명에서 맨 앞 글자만 소문자로 바꾸면 스프링 빈의 기본 이름이 됨
- 전에 AppConfig에서는 @Bean 으로 직접 설정 정보를 작성했고, 의존관계도 직접 명시했으나 AutoAppConfig에는 이런 설정 정보가 없으므로 의존관계 주입 문제를 해결해야 함 → 의존관계를 주입받아야 하는 클래스의 생성자에 @Autowired 사용!
- @Autowired
    - 생성자에 @Autowired를 지정하면 스프링 컨테이너가 자동으로 해당 스프링 빈을 찾아서 주입함
    - 타입이 같은 빈을 찾아서 주입해줌
        - 자식타입이 스프링 빈에 저장되어 있으면 코드에 부모 타입이 있을 때, 후보군이 될 수 있음 근데 여러개이면 충돌이 날 수 있음
- 코드
    - 구현체 클래스에 @Component 붙이기
    - 의존관계 주입이 필요한 곳의 생성자에 @Autowired 붙이기
        - 여러 의존관계도 한번에 주입받을 수 있음

        ```java
        // 의존관계를 자동적으로 해줌
        // 생성자 위에 @Autowired 붙임
        // 컴포넌트 스캔을 쓰게 되면 Autowired를 자동으로 쓰게 됨
        @Autowired
        public MemberServiceImpl(MemberRepository memberRepository) {
            this.memberRepository = memberRepository;
        }
        ```

    - AutoAppConfig

        ```java
        @Configuration
        @ComponentScan(
                // Configuration 어노테이션 붙은 AppConfig, TestConfig 등을 스캔하지 않도록
                // 보통 제거하지 않는데 지금까지 예제 코드를 남기기 위해서 이 방법을 선택함
                excludeFilters = @ComponentScan.Filter(type = FilterType.ANNOTATION, classes = Configuration.class)
        )
        public class AutoAppConfig {
        
        }
        ```

### 💡  탐색 위치

- 모든 자바 클래스를 다 컴포넌트 스캔하면 시간이 오래 걸리므로 필요한 위치부터 탐색하도록 시작 위치를 지정할 수 있음
    - basePackages : 탐색할 패키지의 시작 위치를 지정함, 이 패키지를 포함해서 하위 패키지를 모두 탐색함

        ```java
        @ComponentScan(
                 basePackages = "hello.core",
        }
        ```

    - basePackageClasses : 지정한 클래스의 패키지를 탐색 시작 위치로 지정함
    - 만약 지정하지 않으면 @ComponentScan이 붙은 설정 정보 클래스의 패키지가 시작 위치가 됨
- 패키지 위치를 지정하지 않고, 설정 정보 클래스의 위치를 프로젝트 최상단에 두면 모두 자동으로 컴포넌트 스캔의 대상이 됨
    - 프로젝트 구조가 com.hello, com.hello.service, com.hello.repository 이렇게 되어 있으면 com.hello 가 프로젝트 시작 루트이므로 여기에 AppConfig같은 메인 설정 정보를 두고 @ComponentScan 애노테이션을 붙이고 basePackages 지정은 생략
    - 스프링 부트를 사용하면 @ComponentScan 설정이 들어있는 @SpringBootApplication 를 프로젝트 시작 루트 위치에 두는 것이 관례임

### 💡 컴포넌트 스캔 기본 대상

- 컴포넌트 스캔은 @Component뿐만 아니라 다음 내용도 추가로 대상에 포함함
    - @Component : 컴포넌트 스캔에서 사용
    - @Controller : 스프링 MVC 컨트롤러에서 사용 / MVC 컨트롤러로 인식
    - @Service : 스프링 비지니스 로직에서 사용 / 특별한 처리를 하지 않지만 어노테이션을 보고 개발자들이 여기에 핵심 비지니스 로직이 있음을 인식할 수 있음
    - @Repository : 스프링 데이터 접근 계층에서 사용 / 스프링 데이터 접근 계층으로 인식하고 데이터 계층의 예외를 스프링 예외로 변환해줌
    - @Configuration : 스프링 설정 정보에서 사용 / 스프링 설정 정보로 인식하고 스프링 빈이 싱글톤을 유지하도록 추가 처리를 해줌

  → 이 어노테이션들은 @Component를 포함하고 있음

  → 어노테이션에는 상속 관계가 없는데 어노테이션이 다른 어노테이션을 들고 있는 것을 인식할 수 있는 건 자바가 아닌 스프링이 지원하는 기능 덕분

### 💡 필터

- 코드
    - 포함할 컴포넌트

        ```java
        @Target(ElementType.TYPE)
        @Retention(RetentionPolicy.RUNTIME)
        @Documented
        public @interface MyIncludeComponent {
        }
        
        ```

    - 포함하지 않을 컴포넌트

        ```java
        @Target(ElementType.TYPE)
        @Retention(RetentionPolicy.RUNTIME)
        @Documented
        public @interface MyExcludeComponent {
        }
        ```

    - BeanA
        - 아까 만든 @MyIncludeComponent 어노테이션을 붙여서 컴포넌트 스캔 대상에 추가할 클래스

        ```java
        @MyIncludeComponent
        public class BeanA {
        }
        ```

    - BeanB
        - 아까 만든 @MyExcludeComponent 어노테이션을 붙여서 컴포넌트 스캔 대상에서 제외할 클래스

        ```java
        @MyExcludeComponent
        public class BeanB {
        }
        ```

    - ComponentFilterAppConfigTest
        - includeFilters 를 사용해서 MyIncludeComponent.class를 컴포넌트 스캔 대상을 추가로 지정했기 때문에 BeanA 가 스프링 빈에 등록되었고 excludeFilters를 사용해서 MyExcludeComponent.class를 컴포넌트 스캔에서 제외했기 때문에 BeanB 가 스프링 빈에 등록되지 않음

        ```java
        public class ComponentFilterAppConfigTest {
        
            @Test
            void filterScan() {
                AnnotationConfigApplicationContext ac = new AnnotationConfigApplicationContext(ComponentFilterAppConfig.class);
                BeanA beanA = ac.getBean("beanA", BeanA.class);
                Assertions.assertThat(beanA).isNotNull();
        
                ac.getBean("beanB", BeanB.class);
                org.junit.jupiter.api.Assertions.assertThrows(NoSuchBeanDefinitionException.class,
                        () -> ac.getBean("beanB", BeanB.class));
        
            }
        
            @Configuration
            @ComponentScan(
                    includeFilters = @Filter(type = FilterType.ANNOTATION, classes = MyIncludeComponent.class),
                    excludeFilters = @Filter(type = FilterType.ANNOTATION, classes = MyExcludeComponent.class)
            )
            static class ComponentFilterAppConfig {
        
            }
        }
        ```

- 보통 사용할 때는 @Component면 충분하기 때문에 includeFilters를 사용할 일은 많지는 않음 가끔 컴포넌트 스캔에서 제외하고 싶은 게 있을 땐 excludeFilters를 사용하면 됨
- 최근 스프링 부트에서는 컴포넌트 스캔을 기본으로 제공하므로 옵션을 변경하면서 사용하는 것보다 스프링의 기본 설정에 최대한 맞추어 사용하는 것이 권장됨

### 💡  중복 등록과 충돌

- 컴포넌트 스캔에서 같은 빈 이름이 등록될 경우
    - 자동 빈 등록 vs 자동 빈 등록
    - 수동 빈 등록 vs 자동 빈 등록
- 자동 빈 등록 vs 자동 빈 등록
    - 이 경우에는 스프링이 ConflictingBeanDefinitionException 예외 발생시키면서 오류를 발생시킴
- 수동 빈 등록 vs 자동 빈 등록
    - 수동 빈 등록이 우선권을 가짐 (수동 빈이 자동 빈을 오버라이딩함)
    - 최근 스프링 부트에서는 수동 빈 등록과 자동 빈 등록이 충돌나면 오류가 발생하도록 함
    - spring.main.allow-bean-definition-overriding=true 를 설정하면 수동 빈이 자동 빈을 오버라이딩하도록 함 (디폴트 값은 false)

### 💡 새롭게 알게 된 점 및 중요한 점 정리

1. @Component, @ComponentScan, @Configuration, @Autowired
    - @Component
        - 스프링 빈에 등록하고 싶은 클래스에 @Component 어노테이션을 붙이면 스프링 컨테이너가 자동으로 스프링 빈에 등록함
        - @Service, @Repository, @Controller 등이 @Component에 속함
    - @ComponentScan
        - @Component가 붙은 클래스를 찾아서 자동으로 스프링 컨테이너에 빈으로 등록해주는 설정 클래스에 붙여서 사용하는 어노테이션
        - 설정 클래스에 @Configuration과 함께 쓰임
        - 설정 클래스와 같은 패키지 또는 하위 패키지를 스캔
    - @Configuration
        - 스프링 설정 클래스에 붙여 사용하는 어노테이션, 설정 정보를 담고 있는 클래스
        - @Bean을 이용해 수동으로 빈을 등록할 수 있게 해줌
    - @Autowired
        - 의존 관계 주입이 필요한 객체를 스프링 컨테이너에서 찾아서 자동으로 주입해줌
        - 타입이 같은 빈을 찾아서 주입해줌
        - 보통 컴포넌트 스캔을 쓰게 되면 @Autowired를 자동으로 쓰게 됨
2. 자동 빈 등록 vs 수동 빈 등록
    - 자동 빈 등록 : 설정 클래스에 @Configuration, @ComponentScan을 붙이고 등록할 클래스에 @Component 설정이 들어있는 어노테이션(@Service, @Repository 등)을 붙여서 스프링 컨테이너에 자동으로 등록하는 방식
    - 수동 빈 등록 : 설정 클래스에 @Configuration을 붙이고 해당 클래스 내에서 스프링 빈으로 등록할 메서드에 @Bean을 붙여서 직접 스프링 빈을 등록하는 방식
