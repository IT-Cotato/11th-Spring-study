## 1. 값 타입

- **값 타입 :** 식별자가 없고, 값 자체로만 의미를 가지는 데이터 타입
    - ex : int, Integer, String과 같은 자바의 기본 타입이나 객체
- **엔티티 타입 :** @Entity로 정의, 데이터가 변해도 식별자로 추적 가능
    - ex : 회원 엔티티의 키나 나이 값이 바뀌어도 식별자로 인식
- 값 타입은 변경 시 추적이 불가능
- 값 타입의 분류:
    - **기본값 타입**: int, double 등 자바 기본 타입, Integer, Long 등 래퍼 클래스, String 등
    - **임베디드 타입(복합 값 타입)**: 여러 기본값 타입을 모아 만든 타입
    - **컬렉션 값 타입**: 값 타입의 집합을 저장하는 타입

## 2. 기본값 타입

- 엔티티의 생명주기에 의존
    - ex : 회원을 삭제하면 이름, 나이 필드도 함께 삭제
- 절대 공유하지 않음. 항상 값을 복사
- 래퍼 클래스나 String은 공유 객체이지만 변경 불가능
- ex : String name, int age와 같이 사용함.

## 3. 임베디드 타입(복합 값 타입)

- **임베디드 타입 :** 여러 기본값 타입을 모아 새로운 값 타입을 직접 정의
    - ex: 회원 엔티티가 이름, 근무 시작일, 종료일, 주소(도시, 번지, 우편번호)를 가질 때, 이를 임베디드 타입으로 묶을 수 있음
- 임베디드 타입 사용법:
    - @Embeddable: 값 타입을 정의하는 곳에 표시
    - @Embedded: 값 타입을 사용하는 곳에 표시
    - 기본 생성자 필수
- 임베디드 타입의 장점:
    - 재사용 가능, 높은 응집도, 의미 있는 메소드 추가 가능(예: Period.isWork())
    - 값 타입을 소유한 엔티티의 생명주기에 의존

## 4. 임베디드 타입과 테이블 매핑

- 임베디드 타입은 엔티티의 값일 뿐이며, 테이블 설계와 매핑은 동일
- ORM 애플리케이션에서는 임베디드 타입을 세밀하게 매핑하는 것이 일반적
- 매핑한 테이블의 수보다 클래스의 수가 더 많을 수 있음

## 5. 임베디드 타입과 연관관계, 속성 재정의

- 엔티티에서 같은 값 타입을 여러 번 사용하면 컬럼명이 중복될 수 있음
- @AttributeOverrides, @AttributeOverride를 사용해 컬럼명을 재정의
- 임베디드 타입의 값이 null이면 매핑한 컬럼 값도 모두 null이 됨

## 6. 값 타입과 불변 객체 설계

- 값 타입은 **불변 객체(immutable object)**로 설계하는 것이 안전
    - 생성 시점 이후 값을 변경할 수 없음
    - 생성자로만 값을 설정하고, Setter를 만들지 않음
    - 예시: Integer, String은 자바가 제공하는 대표적인 불변 객체임
- 불변 객체로 만들면 부작용(side effect)을 원천 차단할 수 있음
- 값 타입을 여러 엔티티에서 공유하면 위험. 값 타입은 복사해서 사용해야 함
- 자바의 기본 타입은 값을 복사하지만, 객체 타입은 참조를 전달하므로 공유 참조로 인한 부작용이 발생할 수 있음
    - ex :
        - int a = 10; int b = a; // 값 복사
        - Address a = new Address("Old"); Address b = a; // 참조 전달, b.setCity("New") 시 a도 변경됨
- 작은 제약(불변성)으로 큰 재앙(부작용)을 막을 수 있음

## 7. 값 타입의 비교 방법

- 값 타입은 인스턴스가 달라도 값이 같으면 같은 것으로 봄
    - ex :
        - int a = 10; int b = 10; // 값이 같으면 같음
        - Address a = new Address("서울시"); Address b = new Address("서울시"); // 값이 같으면 같음
- 비교 방법:
    - 동일성(identity) 비교: 인스턴스의 참조 값을 비교, == 사용
    - 동등성(equivalence) 비교: 인스턴스의 값을 비교, equals() 사용
    - 값 타입은 equals() 메소드를 적절히 재정의해야 함

## 8. 컬렉션 값 타입

- 컬렉션 값 타입은 값 타입을 하나 이상 저장할 때 사용
    - @ElementCollection, @CollectionTable 어노테이션 사용
    - 데이터베이스에는 별도의 테이블에 저장함
- 값 타입 컬렉션의 제약사항:
    - 엔티티와 달리 식별자 개념이 없음
    - 변경 사항이 발생하면, 주인 엔티티와 연관된 값을 모두 삭제하고 다시 저장함
    - 테이블은 모든 컬럼을 묶어서 기본 키로 사용함: null 입력X, 중복 저장X
- 실무에서는 상황에 따라 값 타입 컬렉션 대신 일대다 관계를 위한 엔티티를 만들고, 여기서 값 타입을 사용하는 것이 대안이 될 수 있음
    - 전이(Cascade) + 고아 객체 제거를 사용해서 값 타입 컬렉션처럼 사용 가능
    - 예시: AddressEntity

## 9. 엔티티 타입과 값 타입의 정리

- 엔티티 타입 특징:
    - 식별자O, 생명주기 관리, 공유 가능
- 값 타입 특징:
    - 식별자X, 생명주기를 엔티티에 의존, 공유하지 않는 것이 안전(복사해서 사용), 불변 객체로 만드는 것이 안전
- 값 타입과 엔티티 타입을 혼동하지 말아야 하며, 값 타입은 정말 값 타입이 필요할 때만 사용해야 함
- 값 타입을 변경해야 한다면 엔티티로 만들어야 함
