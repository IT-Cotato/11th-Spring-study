# 스프링이란?

스프링  = 여러 기술들의 모음

## 스프링 생태계

1. **스프링 프레임워크***
    1. 자바 백엔드의 강력한 뼈대
    
2. **`스프링 부트*** - 여러 스프링 기술들을 편하게 사용할 수 있게 도와주는 스프링 부트
    1. 장점
        1. 단독 실행 스프링 애플리케이션을 쉽게 생성
        2. 웹 서버가 내장 되어 별도의 웹 서버 설치가 불필요함
            1. 이전엔 빌드를 하고 별도의 웹 서버를 설치하고 다시 빌드하는 등 복잡한 과정이 필요했으나 이젠 스프링 부로 쉽게 가능
        3. 손쉬운 빌드 구성을 위한 starter 종속성 제공
            1. starter : 자동 구성 + 패키징된 필수 라이브러리 모음
                1. ex : spring-boot-starter-web
        4. 스프링과 3rd parth(외부) 라이브러리 자동 구성
            1. 이전엔 외부 라이브러리와 맞는 스프링 버전을 고려했어야 했는데 이젠 스프링이 자동으로 맞춰줌
        5. 메트링, 상태 확인, 외부 구성 같은 프로덕션 준비 기능 제공
            1. 운영 환경에서의 모니터링 기능을 스프링 부트가 제공해줌
        6. 관레에 의한 간결한 설정
            1. 디폴트 설정 및 메뉴얼을 제공해 시작이 쉽고, 필요 시 커스터마이징 가능
    2. 스프링 부트는 스프링 프레임워크와 별도로 사용하는 것이 아님. 스프링 부트는 스프링 프레임 워크를 사용하여 편리함을 제공하는 기능.
    
3. **스프링 데이터**
    1. crud 를 간단하게 할 수 있게 도와줌
    2. Spring data JPA를 가장 많이 씀
    
4. **스프링 세션**
5. **스프링 시규리티**
6. **스프링 Rest Docs**
    1. API 문서랑 테스트를 엮어서 API 문서를 편리하게 해줌
7. **스프링 배치(Batch)**
    1. 배치 처리 : 여러 작업이나 데이터를 한꺼번에 모아뒀다가 정해진 시간 혹은 조건에 맞춰 처리하는 방식
        1. 느리고 실시간 반응이 불가능하지만 대량 처리에 효율적이고 안정적
8. **스프링 클라우드**

## 스프링의 뜻 - 문맥에 따라 다르게 사용됨

1.  스프링 DI 컨테이너 기술 - 스프링 빈 관리
2. 스프링 프레임워크
3. 스프링 부트, 스프링 프레임워크 등을 모두 포함한 스프링 생태계

## 스프링의 핵심

스프링은 자바의 가장 큰 특징인 객체 지향을 살려내는 프레임워크

# 좋은 객체 지향 프로그래밍이란?

## 객체 지향 특징

1. **추상화**
2. **캡슐화**
3. **상속**
4. **다형성** - 객체 지향의 핵심
    1. 유연함과 변경 용이
    2. **역할과 구현으로 구분**하면 세상이 단순해지고, 유연해지며 변경도 편리해진다.
        1. 클라이언트는 대상의 역할(인터페이스)만 알아도 됨
        2. 클라이언트는 구현 대상의 내부 구조를 몰라도 됨
        3. 클라이언트는 구현 대상의 내부 구조 및 대상 자체가 변경되어도 영향을 받지 않음
        4. **클라이언트의 변경 없이 서버 구현 기능을 유연하게 변경할 수 있음 → 확장 가능한 설계**
        5. ex: 운전자(클라이언트) - 자동차(구현 대상)
    3. **인터페이스를 안정적으로 설계하는 것이 매우 중요**
        1. 인터페이스 자체가 변하면 클라이언트, 서버 모두 큰 변경 발생
    4. 자바
        1. 역할 = 인터페이스 
        2. 구현 = 인터페이스를 구현한 클래스, 구현 객체
        3. 객체 설계 시 역할과 구현을 명확히 분리
        4. **객체 설계 시 역할(인터페이스)를 먼저 부여하고, 그 역할을 수행하는 구현 객체 만듦**

## 객체지향 프로그래밍

1. 컴퓨터 프로그램을 객체들의 모임으로 파악하며 각각의 객체는 메시지를 주고 받고 데이터를 처리할 수 있다.
2. 프로그램을 유연하고 변경이 용이하게 만들어 대규모 소프트웨어 개발에 많이 사용된다.(다형성)

## 스프링과 객체지향

1. 스프링은 다형성을 극대화해서 이용할 수 있게 도와줌
2. 스프링에서 제어의 역전(IoC), 의존관계 주입(DI)은 다형성을 활용해서 역할과 구현을 편리하게 다룰 수 있도록 지원
3. 구현의 편리한 변경이 가능해짐

# 좋은 객체 지향 설계의 5가지 원칙(SOLID)

## SRP : 단일 책임 원칙(Single Responsibility Principle)

- 한 클래스는 하나의 책임만 가져야 한다(책임은 문맥과 상황에 따라 다름)
- 중요한 기준은 변경. 변경이 있을 때 파급 효과가 적으면 단일 책임 원칙을 잘 따른 것

## OCP : 개방-폐쇄 원칙(Open/Closed principle)*

- 소프트웨어 요소는 확장에는 열려 있으나 변경에는 닫혀 있어야 한다.
- 다형성 활용
- 클래스의 추가 등의 확장은 괜찮으나 기존 코드의 수정 같은 변경은 일어나지 않아야 된다.
- 문제점
    - 구현 객체를 변경하려면 클라이언트 코드를 변경해야 함→ 다형성을 사용하여도 OCP 원칙이 지켜지지 않음
    - 해결 방법 : 객체를 생성하고, 연관관계를 맺어주는 별도의 조립, 설정자 필요

## LSP : 리스코프 치환 원칙(Liskov Substitution Principle)

- 프로그램의 객체는 프로그램의 정확성을 깨뜨리지 않으면서 하위 타입의 인스턴스로 바꿀 수 있어야 함
    - 규약을 지키는 단계
        - ex : 악셀은 밟으면 앞으로 가야 되는데 뒤로 가게 만든다면? 컴파일 오류는 안 나겠지만 규약을 지키지 않은 것!

## ISP : 인터페이스 분리 원칙(Interface Segregation Principle)

- 특정 클라이언트를 위한 인터페이스 여러 개가 범용 인터페이스 하나보다 나음
    - 자동차 인터페이스 → 운전 인터페이스, 정비 인터페이스로 분리
    - 사용자 클라이언트 → 운전자 클라이언트, 정비사 클라이언트로 분리
    - 분리하면 정비 인터페이스 자체가 변해도 운전자 클라이언트에 영향을 주지 않음
- 인터페이스가 명확해지고, 대체 가능성이 높아짐

## DIP: 의존관계 역전 원칙(Dependency inversion Principle)*

- 추상화에 의존, 구체화엔 의존하지 않음
- ⇒  클라이언트 코드가 구현 클래스에 의존하지 말고, 인터페이스에 의존해야 함
- 역할에 의존해야 한다는 것과 같음. 구현에 의존하는 게 아님

다형성 만으로는 OCP, DIP를 지킬 수 없음. → 뭔가 더 필요함

# 객체 지향 설계와 스프링

- 스프링은 다음 기술로 다형성 + OCP, DIP 가능하게 지원
    - DI(Dependency Injection) : 의존관계, 의존성 주입
    - DI 컨테이너 제공 - 자바 객체들을 어떤 컨테이너 안에 넣어놓고 이 안에서 의존관계를 서로 연결해주고 주입해주는 기능 제
- 클라이언트 코드의 변경 없이 기능 확장

- 인터페이스를 모든 기능에 설계하는 것이 이상적
- 하지만 도입 시 추상화라는 비용이 발생
- 기능 확장 가능성이 없을 시, 구체 클래스를 직접 사용하고, 향후 꼭 필요할 때 리팩터링해서 인터페이스를 도입하는 것도 방법
