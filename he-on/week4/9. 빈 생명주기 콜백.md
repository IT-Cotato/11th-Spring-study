## 스프링 빈의 라이프사이클

1. 스프링 컨테이너 생성
2. 스프링 빈 객체 생성
3. 의존관계 주입
4. 초기화 콜백 호출
    - 초기화 작업은 의존관계 주입이 모두 완료되고 난 다음에 호출해야 되는데 개발자 입장에선 의존관계 주입이 모두 완료된 시점을 알기 어려우므로 콜백 메서드를 통해 초기화 시점을 알려주는 것
5. 빈 사용
6. 소멸전 콜백 호출
7. 스프링 컨테이너 종료

## 객체 생성과 초기화 작업을 분리해야 하는 이유

- 초기화 작업은 외부 커넥션을 연결하거나 무거운 동작을 수행하는 경우가 많음
    
    ⇒ 다른 단계에서 초기화 작업이 같이 진행될 시에 시간이 오래 걸릴 것이고 유지보수도 어려워 질 수 있음
    
- 의존관계 주입이 필요한 초기화 작업이면객체 생성 시점에는 아직 의존 객체가 주입되지 않았을 수도 있음

## 스프링에서 빈의 초기화와 종료를 지원하는 방법

### InitializingBean, Disposable 인터페이스 구현

- InitializingBean
    - afterPropertiesSet() 메서드로 초기화 지원
        - 생성자 호출 후 이 메서드가 호출 됨
- Disposable
    - destroy() 메서드로 소명 지원
        - 컨테이너 종료 시점에 메서드가 호출됨
- 단점 : 스프링 전용
    - 코드가 스프링에 의존하게 됨
    - 초기화/소명 메서드의 이름 변경 불가능
    - 외부 라이브러리처럼 코드를 직접 수정할 수 없는 경우 적용 어렵
    
    ⇒ 스프링 초창기에 많이 사용됐고 지금은 잘 안 씀!
    

### 설정 정보에 초기화 메서드와 종료 메서드를 지정하는 방법

- 특징
    - @Bean 애노테이션에 initMethod와 destroyMethod 속성을 사용해 초기화와 종료 메서드 지정
    - 코드의 수정 없이 설정 정보만으로 초기화/종료 메서드 지정 가능
    - 생성자 호출 후 init 메서드 호출
    - 컨테이너 종료 시점에 close 메서드 호출
- 장점
    - 메서드 이름을 자유롭게 정할 수 있음
    - 스프링 코드에 의존하지 않음
    - 외부 라이브러리에도 적용할 수 있음
- 종료 메서드 추론 기능(@Bean 의 destroyMethod 속성의 기능)
    - 대부분의 라이브러리는 종료 메서드 이름이 close나 shutdown
    - 기본값이 (inferred)로 설정되어 있어 close나 shutdown이라는 이름의 메서드를 자동으로 찾아서 호출해줌(따로 지정 필요 없음)
    - destroyMethod=”” ⇒ 추론 기능 사용하지 않을 때

### @PostConstruct, @PreDestroy 애노테이션 사용

- 최신 스프링에서 가장 권장하는 방식
- 초기화 메서드 위에 @PostConstruct, 종료 메서드 위에 @PreDestroy 애노테이션 붙여주기
- 특징
    - 스프링 종속 기능이 아니라 자바 표준이라 스프링이 아닌 다른 컨테이너에서도 동작
    - 컴포넌트 스캔과 함께 사용할 때도 아주 편리
- 단점
    - 코드를 고칠 수 없는 외부 라이브러리에는 적용하기 어려움

**일반적으로는 @PostConstruct, @PreDestroy 애노테이션 사용**

**외부 라이브러리를 초기화하고 종료해야 될 때는 @Bean의 initMethod, destroyMethod 속성 사용**
