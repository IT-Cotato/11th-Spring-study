## 1. 프록시 개념과 특징

- **프록시 : 실제 엔티티 객체 대신 사용하는 가짜 객체**
    - 데이터베이스 조회를 미루고, 실제 데이터가 필요할 때만 조회
    - em.find() : 실제 엔티티를 즉시 조회
    - em.getReference() : 프록시 객체를 반환하여 실제 조회를 미룸
- 프록시 객체는 실제 엔티티를 상속받아 만들어지므로, 겉모습은 같지만 **실제 객체와 구분이 필요**
    - **타입 체크 시** == 비교는 실패할 수 있으므로, **instance of를 사용**해야 함
- 프록시 객체는 실제 엔티티의 참조(target)를 보관하고, 메소드 호출 시 실제 엔티티의 메소드를 호출함
- 프록시는 처음 사용할 때 한 번만 초기화되며, 초기화 후에는 실제 엔티티에 접근 가능함
- 영속성 컨텍스트에 엔티티가 이미 있으면 em.getReference()도 실제 엔티티를 반환
- 프록시 초기화 여부는 PersistenceUnitUtil.isLoaded(Object entity)로 확인 가능하며, 프록시 클래스 이름을 통해서도 확인 가능
    - 강제 초기화는 org.hibernate.Hibernate.initialize(entity) 또는 프록시의 메소드 호출로 가능
    - JPA 표준에는 강제 초기화 기능이 없음

## 2. 즉시 로딩과 지연 로딩

- **즉시 로딩(EAGER)**: 연관된 엔티티를 함께 즉시 조회함. 예를 들어, Member를 조회할 때 Team도 함께 조회
    - @ManyToOne, @OneToOne은 기본이 즉시 로딩임. 하지만 실무에서는 LAZY로 설정하는 것이 좋음
    - 즉시 로딩은 예상치 못한 SQL이 발생할 수 있고, JPQL에서 N+1 문제를 일으킴
- **지연 로딩(LAZY)**: 연관된 엔티티를 프록시로 조회하고, 실제로 사용할 때 DB에서 조회
    - @ManyToOne(fetch = FetchType.LAZY)로 설정 가능함
    - Team을 프록시로 조회하고, team.getName() 등 실제 사용 시점에 DB 조회가 일어남
- **실무에서는 모든 연관관계에 지연 로딩을 사용하는 것이 권장됨**
    - 즉시 로딩 대신 JPQL fetch 조인이나 엔티티 그래프 기능을 활용하는 것이 좋음

## 3. 지연 로딩 활용 전략

- 연관관계의 사용 빈도에 따라 로딩 전략을 선택해야 함
    - Member와 Team처럼 자주 함께 사용하는 경우: 즉시 로딩
    - Member와 Order처럼 가끔 함께 사용하는 경우: 지연 로딩
    - Order와 Product처럼 자주 함께 사용하는 경우: 즉시 로딩
- 하지만 **실무에서는 모든 연관관계에 지연 로딩을 적용하고, 필요한 경우 fetch 조인 등으로 최적화하는 것이 바람직**

## 4. 영속성 전이(CASCADE)

- **영속성 전이 : 부모 엔티티를 영속 상태로 만들 때, 연관된 자식 엔티티도 함께 영속 상태로 만드는 기능**
    - ex : 부모 엔티티를 저장할 때 자식 엔티티도 함께 저장됨 .
    - 설정 : @OneToMany(mappedBy="parent", cascade=CascadeType.PERSIST)
- 영속성 전이는 연관관계를 매핑하는 것과는 별개로, 엔티티를 함께 영속화하는 편리함 제공
- CASCADE의 종류:
    - ALL: 모두 적용
    - PERSIST: 영속
    - REMOVE: 삭제
    - MERGE: 병합
    - REFRESH: 새로고침
    - DETACH: 분리

## 5. 고아 객체와 제거

- **고아 객체 : 부모 엔티티와 연관관계가 끊어진 자식 엔티티**
    - orphanRemoval = true로 설정하면, 부모와의 연관관계가 끊어진 자식 엔티티를 자동 삭제
    - ex : parent1.getChildren().remove(0); 실행 시 자식 엔티티가 삭제
    - 실제로는 DELETE FROM CHILD WHERE ID=? 쿼리 실행
- 고아 객체 제거 기능은 참조하는 곳이 하나일 때 사용해야 하며, 특정 엔티티가 개인 소유할 때 적합
    - @OneToOne, @OneToMany에서만 사용 가능
    - 부모를 제거하면 자식도 고아가 되어 함께 제거됨. 이는 CascadeType.REMOVE와 유사하게 동작

## 6. 영속성 전이 + 고아 객체, 생명주기 관리

- **CascadeType.ALL + orphanRemoval=true를 함께 사용하면, 부모 엔티티를 통해 자식 엔티티의 생명주기를 관리할 수 있음**
    - 부모 엔티티를 em.persist()로 영속화하거나 em.remove()로 제거할 때, 자식 엔티티도 함께 처리됨
    - 도메인 주도 설계(DDD)에서 Aggregate Root 개념을 구현할 때 유용함
