# 스프링 컨테이너

- ApplicationContext : 인터페이스, 스프링 컨테이너
- 스프링 컨테이너는 XML 기반으로 만들수 있고, 애노테이션 기반의 자바 설정 클래스로 만들 수 있음
    - Bean 등록 및 의존성 주입 방법이 두 가지로 나뉜다는 뜻
        - XML :
            - 과거에 많이 쓰던 방법
            - 수작업 많이 필요
            - 더 유연하지만 더 복잡함
            - 설정 위치 : 외부 XML 파일
            - 비유 : 직접 설계도 그리고, 일일이 가구 배치까지 다 하는 인테리어
        - 애노테이션 :
            - 요즘 많이 쓰임
            - 자동화됨
            - 개발자 입장에서 더 편하고 간단함
            - 빠르고 간결하지만 시스템에 많이 의존함 → 유연성 부족
            - 비유 :  자동으로 가구 배치해주는 인공지능 인테리어 시스템
            
- AppConfig 방식이 애노테이션 기반의 자바 설정 클래스로 스프링 컨테이너를 만든 것

- 스프링 컨테이너를 얘기 할 때 다음으로 구분해서 이야기 함
    - **BeanFactory** → 직접 사용할 일 거의 없음
        - 스프링 컨테이너 최상위 인터페이스
        - 스프링 빈 관리 조회 역할
        - getBean() 제공
    - **ApplicationContext** → 일반적으로 얘를 스프링 컨테이너라고 함
        - BeanFactory 기능을 모두 상속 받아 제공 + 편리한 부가기능 제공
        - 부가 기능
            - 메세지 소스 → 국제화
                - 들어오는 국가의 언어로 출력해 줌 (한국에서 들어오면 한국어)
            - 환경 변수
                - 로컬, 개발, 운영 등 구분 처리
            - 애플리케이션 이벤트
                - 이벤트 발행, 구독 모델 편리하게 지원
            - 편리한 리소스 조회
                - 파일, 클래스패스, 외부 등에서 리소스 편리하게 조회

# 스프링 컨테이너 생성 과정

1. 스프링 컨테이너 생성
    1. AppConfig.class 로 스프링 컨테이너가 생성됨
        1. 컨테이너 안엔 스프링 빈 저장소의 구성정보가 지정됨
    2. AppConfig.class 의 구성정보를 활용해 a-i가 이루어짐
2. 스프링 빈 등록
    1. 스프링 컨테이너는 파라미터로 넘어온 설정 클래스 정보를 사용해 스프링 빈 등록
    - 빈 이름
        - 빈 이름은 기본적으로 메소드 이름을 사용함
        - 직접 부여하고 싶다면
            - @Bean (name = “memberService2”)
        - 빈 이름은 항상 다른 이름을 부여해야 함
3. 스프링 빈 의존관계 설정
    1. 스프링 컨테이너는 설정 정보를 참고해 의존 관계 주입(DI)
- 자바 코드로 스프링 빈 등록 시 2,3번이 한번에 처리 됨

# 빈 출력

- 스프링에 등록된 모든 빈 이름 조회
    
    → ac.getBeanDefinitionNames()
    
- 빈 이름으로 빈 객체(인스턴스) 조회
    
    → ac.getBean(빈 이름, 타입)
    
    → ac.getBean(타입)
    
    - 타입으로 조회 시 같은 타입의 스프링 빈이 둘 이상이면 오류 발생
        
        → 빈 이름 지정
        
    - ac.getBeansOfType() 사용시 해당 타입의 모든 빈 조회 가능
- 애플리케이션 빈 - 내가 등록한 빈만 출력
    
    → ROLE_APPLICATION
    
- 애플리케이션 빈 - 스프링이 내부에서 사용하는 빈
    
    → ROLE_INFRASTRUCTURE
    

! 조회 대상 스프링 빈이 없으면 예외 발생
 → NoSuchBeanDefinitionException: No bean named 'xxxxx' available

스프링 빈 조회  - 상속관계

- 부모 타입으로 조회 시 자식 타입도 함께 조회
    
    → Object(모든 자바 객체의 최고 부모 타입) 으로 조회 시 모든 스프링 빈 조회

# 스프링 빈 설정 메타 정보 - BeanDefinition

- 스프링이 다양한 설정 형식을 지원할 수 있는 이유 ⇒ BeanDefinition이라는 추상화!
- 역할과 구현을 개념적으로 나눈 것
    - 구현이 XML와 자바 코드인 것.
    - 스프링 컨테이너는 구현이 뭐로 되는 지 몰라도 됨. 오직 BeanDefinition만 알면 됨!
- BeanDefinition : 빈 설정 메타정보
    - @Bean, <bean>당 각각 하나씩 메타 정보가 생성됨
- 스프링 컨테이너는 이 메타 정보를 기반으로 스프링 빈 생성
- 정보
    - BeanClassName: 생성할 빈의 클래스 명(자바 설정 처럼 팩토리 역할의 빈을 사용하면 없음)
    - factoryBeanName: 팩토리 역할의 빈을 사용할 경우 이름, 예) appConfig
    - factoryMethodName: 빈을 생성할 팩토리 메서드 지정, 예) memberService
    - Scope: 싱글톤(기본값)
    - lazyInit: 스프링 컨테이너를 생성할 때 빈을 생성하는 것이 아니라, 실제 빈을 사용할 때 까지 최대한 생성을 지연 처리 하는지 여부
    - InitMethodName: 빈을 생성하고, 의존관계를 적용한 뒤에 호출되는 초기화 메서드 명
    - DestroyMethodName: 빈의 생명주기가 끝나서 제거하기 직전에 호출되는 메서드 명
    - Constructor arguments, Properties: 의존관계 주입에서 사용한다. (자바 설정 처럼 팩토리 역할의 빈을 사용하면 없음)
