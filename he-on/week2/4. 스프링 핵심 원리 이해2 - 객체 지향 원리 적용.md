# 할인 정책 변경

할인은 실무에서 테스트를 정말 많이 해야 하는 부분…

할인만 하는 인터페이스를 따로 구현하는 게 매우 중요

만약 안 됐다면? 악덕 기획자가 갑자기 할인 정책 바꿨을 때 수정할 부분이 매우 많아짐…

기존 코드는 DIP를 충족하진 못 했음

- discount interface인 DiscounPolicy(추상 인터페이스)만 의존하고 있는 것이 아니라 FixDiscountPolicy (구체 클래스)도 함께 의존하고 있음
    - 그래서 구체 클래스를 변경할 때 클라이언트 코드(OrderServiceImpl)도 변경해야 했음
- 정책을 변경한 지금도 추상 인터페이스(DiscountPolicy)와 구체 클래스(RateDiscountPolicy)를 함께 의존하고 있음
- 해결방법
    - 클라이언트 코드가 인터페이스에만 의존하도록 설계를 변경해야 함
        - 이렇게 하면 인터페이스 내 어떤 클래스를 사용해야 될 지 알지 못 해 오류가 남
            - **클래스 선정자(AppConfig)를 따로 만들어줘야 함!**
                - 애플리케이션 전체의 동작 방식을 구성(config)하기 위해, 구현 객체를 생성하고, 연결하는 책임을 가지는 별도의 설정 클래스

---

# AppConfig

- 애플리케이션의 실제 동작에 필요한 구현 객체 생성
    - **좋은 AppConfig :**
        - 역할에 따른 구현이 잘 보임
        - 역할과 구현이 명확히 분리됨
        - 중복 제거
- 생성한 객체 인스턴스의 참조(레퍼런스)를 생성자를 통해서 주입(연결)
    - **클라이언트 입장에서는 의존관계를 외부에서 주입해주는 것 같기에 DI(Dependency Injection : 의존관계 주입)이라고 함**

---

# SRP, DIP, OCP 적용

- SRP(단일 책임 원칙) : 한 클래스는 하나의 책임만 가져야 한다
    - AppConfig가 구현 객체 생성 및 연결
    - 클라이언트 객체는 실행
- DIP(의존관계 역전 원칙)
    - 클라이언트 코드가 추상화 인터페이스만 의존
    - AppConfig가 객체 인스턴스를 클라이언트 코드 대신 생성해 클라이언트 코드에 의존관계 주입
- OCP(개방 - 폐쇄 원칙)
    - 애플리케이션을 사용 영역과 구성 영역으로 나눔
    - AppConfig가 의존 관계를 주입하므로 클라이언트 코드는 변경하지 않아도 됨
    - 소프트웨어 요소를 새롭게 확장해도 사용 영역의 변경이 닫혀있음

# 제어의 역전 IoC(Inversion of Control)

- 기존 프로그램 :
    - 구현 객체가 스스로 필요한 서버 구현 객체 생성, 연결, 실행 → 구현 객체가 프로그램의 제어 흐름 조종
- AppConfig 이후 프로그램 :
    - 구현 객체 → 자신의 로직만 실행
    - AppConfig →프로그램 제어 흐름 조종, 구현 객체 생성 및 연결
- **제어의 역전 : 기존 프로그램처럼 구현 객체가 프로그램의 제어 흐름을 직접 조종하는 것이 아닌, 외부에서 관리하는 것**

# 프레임워크 vs 라이브러리

- 프레임워크
    - 내가 작성한 코드를 제어하고 대신 실행
    - 비유 :
        - 프레임워크 = 연극무대
            
            → 무대(프레임워크)가 준비돼 있고, 나는 역할만 맡아 연기함(전체 연출은 무대가)
            
    - ex:JUnit
- 라이브러리
    - 내가 작성한 코드가 직접 제어의 흐름 담당
    - 비유 :
        - 라이브러리 = 공구 상자
            
            → 내가 원하는 도구(라이브러리)를 직접 꺼내서 쓰는 것(전체 작업은 내가 함)

# 의존관계 주입 DI (Dependency Injection)

- 의존관계는 정적인 클래스 의존관계와, 동적인 객체(인스턴스) 의존관계 둘을 분리해서 생각
    - 정적인 클래스 의존관계
        - 클래스가 사용하는 import코드로 의존관계 판단 가능
        - 하지만 실제로 어떤 객체가 주입될 지 모름
    - **동적인 객체 인스턴스 의존 관계**
        - 애플리케이션 실행 시점에 실제 생성된 객체 인스턴스의 참조가 연결된 의존 관계
        - 의존관계 주입 : 실행 시점(런타임)에 외부에서 실제 구현 객체를 생성하고 클라이언트에 전달해서 클라이언트와 서버의 실제 의존관계가 연결되는 것
            - 객체 인스턴스 생성 → 참조값 전달 ⇒ 연결
            - 클라이언트 코드 변경 X, 클라이언트가 호출하는 대상의 타입 인스턴스 변경
            - 정적인 클래스 의존관계를 변경하지 않고, 동적인 객체 인스턴스 의존관계를 쉽게 변경 가능

# IoC 컨테이너(DI 컨테이너)

- AppConfig 처럼 객체를 생성하고 관리하며 의존관계를 연결해주는 것
- 의존관계 주입에 초점을 맞춰 최근에는 DI 컨테이너라고 주로 불림
- 어셈블러, 오브젝트 팩토리 등으로 불리기도 함

---

# 스프링 컨테이너

- ApplicationContext ⇒ 스프링 컨테이너
- 기존의 AppConfig 역할을 스프링 컨테이너를 통해 할 수 있음
- **스프링 빈** :
    - 스프링 컨테이너는 @Configuration 이 붙은 AppConfig 를 설정(구성) 정보로 사용.
    - @Bean 이라 적힌 메서드를 모두 호출해 반환된 객체를 스프링 컨테이너에 등록
    - 이렇게 등록된 객체를 스프링 빈이라고 함
    - @Bean이 붙은 메서드 명을 스프링 빈의 이름으로 사용
    - applicationContext.getBean() 메서드를 사용해 스프링 빈을 찾을 수 있음
    - 이제 스프링 컨테이너에 객체를 스프링 빈으로 등록하고, 스프링 컨테이너에서 스프링 빈을 사용하면 됨
- 스프링 컨테이너의 장점
    - 스프링 부트 3.1부터는 기본 로그 레벨을 INFO로 빠르게 설정하기 때문에 로그를 확인할 수 없는데, 스프링 컨테이너를 사용하면 기본 로그 레벨을 DEBUG로 설정해 로그 확인이 가능해짐
