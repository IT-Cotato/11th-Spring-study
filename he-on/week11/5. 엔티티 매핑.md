## 1. 엔티티와 기본 개념

- **엔티티(Entity)**는 JPA가 관리하는 클래스, 데이터베이스의 테이블과 매핑
- 엔티티 클래스는 반드시 `@Entity` 어노테이션이 붙어야 함
- 엔티티는 **기본 생성자(파라미터 없는 public 또는 protected)**가 필요하며, final 클래스, enum, interface, inner 클래스는 사용할 수 없음
- 저장할 필드에는 `final` 사용 금지
- `@Entity`의 속성 `name`은 JPA에서 사용할 엔티티 이름을 지정, 기본값은 클래스 이름
- `@Table`은 엔티티와 매핑할 테이블을 지정, 이름, 카탈로그, 스키마, 유니크 제약조건 등을 설정 가능

## 2. 엔티티의 자동 생성 및 매핑 요구사항

- **데이터베이스 스키마 자동 생성**은 애플리케이션 실행 시점에 DDL을 자동으로 만들어주는 기능
- 옵션: `create`, `create-drop`, `update`, `validate`, `none`이 있으며, 각각의 용도와 사용 시 주의점이 있음
- 운영 서버에서는 `create`, `create-drop`, `update` 사용 금지, `validate` 또는 `none` 권장
- 유니크 제약조건은 `@Table(uniqueConstraints=...)`로 설정 가능
- DDL 생성 시 필수 조건이나 제약조건을 추가할 수 있으며, 예를 들어 회원 이름은 필수(`nullable=false`), 유니크 제약조건 등

## 3. 필드와 컬럼 매핑

- 엔티티 필드는 `@Column`으로 컬럼에 매핑하며, 이름, 길이, null 허용 여부, 유니크, 기본값 등을 지정 가능
- 날짜 타입 필드에는 `@Temporal`을 사용하며, `TemporalType.DATE`, `TIME`, `TIMESTAMP`로 구분
- `@Enumerated`는 자바 enum 타입을 매핑할 때 사용하며, `STRING` 또는 `ORDINAL`로 저장 방식 선택
- `@Lob`은 큰 데이터(BLOB 또는 CLOB)를 저장할 때 사용
- `@Transient`는 데이터베이스에 저장하지 않고, 메모리상에서만 사용하는 필드에 붙임

## 4. 키(Key) 매핑 방법

- 기본 키는 `@Id`로 지정하며, `@GeneratedValue`로 키 생성 전략을 설정
- 전략 종류:
    - **IDENTITY**: 데이터베이스에 위임, MySQL, PostgreSQL 등에서 사용, `AUTO_INCREMENT`와 유사
    - **SEQUENCE**: 시퀀스 오브젝트 사용, Oracle, PostgreSQL 등에서 유용, `@SequenceGenerator` 필요
    - **TABLE**: 별도 테이블을 만들어 키를 생성, 모든 DB에서 사용 가능, `@TableGenerator` 필요
    - **AUTO**: 방언에 따라 자동 선택, 기본값
- 각 전략의 특징과 사용법, 예제 코드 포함

## 5. 키 생성 전략별 특징과 사용법

- **IDENTITY**: 데이터베이스가 키를 생성, `persist()` 시점에 바로 SQL 실행 후 키 조회 가능
- **SEQUENCE**: 시퀀스 오브젝트를 사용, `@SequenceGenerator`로 이름, 초기값, 증가값 지정
- **TABLE**: 별도 테이블(`MY_SEQUENCES`)을 만들어 키 생성, 성능은 낮지만 모든 DB에서 호환 가능
- 자연키 대신 대리키(대체키)를 사용하는 것이 권장됨 → 자연키는 찾기 어렵고 변할 수 있기 때문

## 6. 실전 예제와 요구사항 분석

- 회원, 주문, 상품 등 도메인 모델 설계와 관계 설정
- 회원은 여러 번 주문 가능하며, 주문과 상품은 다대일 관계로 설계
- 주문할 때 여러 상품을 선택할 수 있으며, 주문 상품 모델로 다대일 관계를 풀어냄
- 객체 설계 시 외래키를 객체에 바로 참조하는 방식의 문제점도 언급됨

## 7. 데이터베이스 설계와 객체 설계의 차이

- 현재 방식은 테이블 설계에 맞춘 객체 설계로, 외래키를 객체에 탐색하는 것이 불가능하여 문제점이 있음
- 객체 설계는 그래프 구조로, 테이블 설계와 차이가 있어 이를 고려한 설계 필요
