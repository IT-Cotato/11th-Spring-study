## 1. 연관관계 매핑의 기본 개념

- 연관관계 매핑 : 객체와 테이블 사이의 관계를 정의하는 것
- 주요 목표
    - 객체와 테이블의 연관관계 차이, 객체의 참조와 테이블의 외래 키 매핑, 방향(단방향/양방향), 다중성(다대일, 일대다, 일대일, 다대다), 연관관계의 주인 개념을 이해하는 것
- 연관관계 매핑에서 **연관관계의 주인(Owner)** 개념 중요
    - 양방향 연관관계에서는 반드시 주인을 지정해야 함
    

## 2. 연관관계가 필요한 이유

- 객체지향 설계의 목표 : **자율적인 객체들의 공동체**를 만드는 것
- 객체들은 서로 협력하며 동작해야 하므로, 객체 간의 연관관계가 필요
- ex
    - 회원과 팀의 관계에서, 회원은 하나의 팀에만 소속될 수 있는 다대일(N:1) 관계
    

## 3. 객체와 테이블의 연관관계 차이

- 객체를 테이블에 맞추어 모델링하면, 참조 대신 외래 키를 직접 사용하게 됨
    - ex: Member 엔티티에 teamId(Long) 필드를 두고, Team 엔티티의 id와 연결
- 외래 키 식별자를 직접 다루는 방식은 객체지향적이지 않음
    - ex: 회원을 조회할 때, teamId로 다시 Team을 조회해야 함
- 객체는 참조(객체 그래프)를 사용하지만, 테이블은 외래 키로 조인함
    - 이로 인해 객체와 테이블 사이에 **큰 간격**이 생김
    

## 4. 단방향 연관관계 매핑

- 객체지향 모델링에서는 객체의 참조와 테이블의 외래 키를 매핑
    - ex: Member 엔티티에 Team 타입의 team 필드를 두고, @ManyToOne, @JoinColumn(name = "TEAM_ID")로 매핑
- 연관관계 저장 시, Member 객체의 team 필드에 Team 객체를 할당
- 연관관계 조회 시, Member 객체의 getTeam() 메서드로 Team 객체를 바로 탐색 가능
- 연관관계 수정도 객체의 참조를 바꾸는 방식으로 처리

## 5. 양방향 연관관계와 연관관계의 주인

- 양방향 매핑에서는 두 객체가 서로를 참조함
    - ex: Member 엔티티는 Team을 참조, Team 엔티티는 List를 참조
- 양방향 탐색이 가능해져, Team에서 소속된 Member 목록을 바로 조회할 수 있음
- **연관관계의 주인(Owner)** 개념 중요
    - 두 객체 중 외래 키가 있는 쪽(Member.team)이 주인이 됨
    - 주인만이 외래 키를 관리(등록, 수정)할 수 있고, 반대편은 읽기만 가능
    - 주인은 mappedBy 속성을 사용하지 않음, 반대편은 mappedBy로 주인을 지정
- 실수 예시: 주인이 아닌 쪽(Team.members)만 값을 설정하면, 외래 키가 null로 저장됨
- 올바른 방법: 주인(Member.team)과 반대편(Team.members) 모두 값을 설정해야 함

## 6. 양방향 연관관계 매핑 시 주의사항

- 항상 양쪽에 값을 설정해야 순수 객체 상태와 DB 상태가 일치함
- 연관관계 편의 메서드를 만들어 두 객체의 관계를 동시에 설정하는 것이 좋음
- 양방향 매핑 시 무한 루프에 주의해야 함
    - ex: toString(), lombok, JSON 생성 라이브러리 등에서 발생
- 단방향 매핑만으로도 연관관계 매핑은 완료됨. 양방향 매핑은 반대 방향 탐색이 필요할 때 추가하면 됨

## 7. 연관관계의 주인 선정 기준

- 연관관계의 주인은 **외래 키의 위치**를 기준으로 정해야 함
- 비즈니스 로직상 중요한 객체가 아니라, 외래 키가 있는 쪽이 주인이 되어야 함
