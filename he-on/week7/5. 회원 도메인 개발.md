### 회원 리포지토리

- 역할
    - 데이터베이스와 직접 상호작용하여 회원의 데이터 관리
        
         ⇒ 회원 저장, 탐색 등의 데이터 작업 전담
        
- @Repository 어노테이션
    - 스프링에게 해당 클래스가 리포지토리 역할을 수행함을 알려주는 어노테이션
    - 스프링 빈 등록, JPA 예외를 스프링 기반 예외로 변환하는 기능 또한 포함
- @PersistenceContext 어노테이션
    - JPA의 EntityManager 주입 받을 때 사용
        - EntityManager = 실제로 데이터베이스 작업을 수행하는 핵심 객체
- 주요 메서드
    - save -회원을 데이터베이스에 저장
    - findOne - 회원 ID로 특정 회원 조회
    - findAll - 모든 회원을 리스트로 가져옴
    - findByName - 이름으로 회원 검색 시 사용

### 회원 서비스

- 역할
    - 비즈니스 로직 처리
        
        ⇒ 리포지토리를 이용해 데이터를 가져오거나 저장하면서 회원 가입이나 회원 목록 조회 같은 핵심 기능을 구현
        
- @Service 어노테이션
    - 이 클래스가 서비스 역할을 한다는 것을 스프링에게 알려주는 어노테이션
- @Transactional 어노테이션
    - 트랜잭션 관리
        
        → 데이터 변경이 일어나는 작업에 필수적
        
        - 트랜잭션
            - 데이터베이스 작업의 최소 단위
            - 중간에 하나라도 실패하면 모든 작업을 취소
                
                ⇒ 데이터 일관성 유지
                
    - @Transactional(readOnly = true)
        - 데이터 변경이 없는 읽기 전용 메서드에 사용
            
            ⇒ 영속성 컨텍스트를 플러시 하지 않기 때문에 성능 향상 기대 가능
            
            ⇒ 데이터베이스 드라이버가 지원하면 DB에서도 성능 향상
            
- 중복 회원 검증 로직 구현 방법
    - validateDuplicateMember 메서드에서 이름으로 회원을 조회해 만약 같은 이름의 회원이 존재할시 예외 발생
    - 실무에서는 멀티 쓰레드 상황을 고려해 회원 테이블의 회원명 칼럼에 유니크 제약 조건을 추가하는 것이 안전
        - 멀티 쓰레드 : 하나의 프로세스 안에서 여러 개의 작업 단위가 동시에 실행되는 구조
        - 즉, 동시에 같은 이름의 회원이 가입을 시도할 경우 둘 다 통과될 가능성이 있음
        - 그렇기에 데이터베이스 레벨에서 유니크 제약 조건을 걸어두면 안전
            - 유니크 제약 조건 : 데이터베이스에서 특정 컬럼의 값이 중복되지 않도록 강제하는 제약 조
    - 필드 주입 대신 생성자 주입 사용
        - 생성자 주입의 장점
            - 객체 생성 시점에 의존성 주입이 완료되므로 final 키워드를 사용할 수 있어 변경 불가능한 안전한 객체 생성 가능
            - 순환 참조 문제를 컴파일 시점에서 잡아낼 수 있음
            - 테스트 코드 작성 시에도 용이
        - @RequiredArgsConstructor
            - Lombok 제공 어노테이션
            - final이 붙은 필드나 @NonNull이 붙은 필드에 대해 생성자를 자동으로 만들어 줌
