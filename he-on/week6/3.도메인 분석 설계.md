## 엔티티 클래스 개발

- 개발 전 제거해야 하는 코드
    - src 하위:
        - jpabook.jpashop.Member
        - jpabook.jpashop.MemberRepository
    - test 하위:
        - jpabook.jpashop.MemberRepositoryTest
- 실무에선 Getter만 열어두고 Setter는 꼭 필요한 경우에만 사용하는 것이 좋음
    - Setter 호출 시 데이터가 변하기 때문에 열어둘 경우 엔티티의 변경 추적이 어려워짐
        - ⇒ 엔티티 변경 시 Setter 대신 변경 지점이 명확하도록 변경을 위한 메서드 별도 제공 필요
- 실무에서 @ManyToMany 사용 X
    - 중간 테이블에 컬럼 추가가 불가능하고 세밀한 쿼리 실행이 어렵기 때문에 실무에선 사용이 어려움
    - 중간 엔티티를 만들고 @ManyToOne, @OneToMany로 매칭해서 사용
    - `다대다 매핑 -> 일대다 매핑, 다대일 매핑으로 풀어서 사용
- 값 타입은 변경 불가능하게 설계
    - @Setter 제거, protected 로 설정

## 엔티티 설계시 주의점

- 엔티티에는 Setter 사용 x
- 실무에서 모든 연관관계는 지연 로딩(LAZY)으로 설정
    - 즉시 로딩(EAGER)은 예측 어렵, SQL 실행 추적 어렵
        - JPAL 실행 시 N+1 문제 자주 발생
    - 연관된 엔티티를 함께 DB에서 조회해야 하면, fetch join 또는 엔티티 그래프 기능 사용
    - @XToOne(OneToOne, ManyToOne) 관계는 기본이 즉시로딩이므로 직접 지연로딩으로 설정 필요
- 컬렉션은 필드에서 초기화
    - null 문제에서 안전
    - 하이버네이트는 엔티티를 영속화 할 때, 컬랙션을 감싸서 하이버네이트가 제공하는 내장 컬렉션으로 변경
    - 만약  `getOrders()` 처럼 임의의 메서드에서 컬력션을 잘못 생성하면 하이버네이트 내부 메커니즘에 문제가 발생 가능
        
        ⇒필드레벨에서 생성하는 것이 가장 안전하고, 코드도 간결
        
- 테이블, 칼럼명 생성
    - 자바에서 만든 변수 이름을 데이터베이스 칼럼명으로 어떻게 바꾸는지
        - 스프링 부트가 자동으로  DB 스타일로 바꿔줌
            - memberPoint → member_point
