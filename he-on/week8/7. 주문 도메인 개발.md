### 도메인 모델 패턴

- 엔티티가 비즈니스 로직을 가지고 객체 지향의 특성을 적극 활용하는 것
- 복잡한 도메인 로직에 적합
- 테스트와 유지보수에 유리하지만 구조가 복잡해질 수 있음
    - 책임의 분산과 협력 구조가 많아짐
        - ⇒ 개발자가 객체 간의 역할, 상태 변화, 협력 순서를 모두 설계하고 고려해야 함

### 트랜잭션 스크립트 패턴

- 엔티티에는 비즈니스 로직이 거의 없고 서비스 계층에서 대부분의 비즈니스 로직을 처리하는 것
- 구현이 단순하고 빠르지만 로직이 분산되어 유지 보수가 힘들어질 수 있음
    - 로직 분산
        - 엔티티가 데이터를 갖고만 있고 모든 판단과 계산을 서비스에서 수행하기 때문에 서비스가 너무 많은 로직을 갖게 되며 하나의 기능이 여러 클래스에 분산
        - 재사용 가능한 단위 로직이 없어 동일한 검증 로직을 여러 곳에서 재작성 해야 함
- CRUD 중심 시스템에 적합

## 동적 쿼리 처리

- 주문 검색은 검색 조건에 따라 쿼리가 동적으로 바뀌기 때문에 동적 쿼리 처리가 중요함
    - 동적 쿼리 처리 방법
        - JPQL을 문자로 직접 생성
            - 코드 직관적
            - 검색 조건이 많아지면 코드 복잡해짐
            - 문자열이기 때문에 컴파일 시점에서 오류 잡기 어려움
            - 유지보수 불편
        - JPA Criteria 사용
            - JPA 표준 스펙으로 동적 쿼리를 안전하게 구성할 수 있도록 도와줌
            - 코드가 너무 복잡하고 장황해짐
            - 실무에선 잘 사용되지 않음
        - Querydsl 사용
            - 가장 효과적
