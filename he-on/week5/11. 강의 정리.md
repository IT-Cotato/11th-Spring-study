## 1장

스프링의 핵심 = 객체 지향의 특징을 살려내는 프레임워크

- 객체 지향 특징
    - 추상화, 캡슐화, 상속, **다형성**
- SOLID - 좋은 객체 지향 설계의 5가지 원칙
    - SRP - 클래스 단일 책임
    - **OCP** - 확장엔 개방, 변경은 폐쇄
    - LSP - 객체는 프로그램의 정확성 보존, 하위 타입의 인스턴스로 변경 가능
    - ISP - 인터페이스 분리
    - **DIP** - 추상화(인터페이스)에 의존, 구체화(클래스)엔 의존 X
- 객체 지향 설계와 스프링
    - DI(의존관계 주입), DI 컨테이너 제공(컨테이너 안의 자바 객체들에게 의존관계 주입 및 연결)

## 2,3강

요구사항을 바꿀 때 생기는 문제 해결을 위해 객체 지향이 제공하는 다형성만을 활용하는 것이 아닌 DI 컨테이너나 의존관계 주입 등을 통해 객체 지향을 제대로 활용

⇒ 스프링이 제공해줌

## 4장

- 스프링 컨테이너(일반적으로 ApplicationContext)
    - 객체(스프링 빈) 생성, 스프링 빈 관리 조회, 국제화, 환경변수, 리소스 조회
- 스프링 컨테이너 생성 방법
    - 스프링 컨테이너 생성
    - 빈 클래스 스캔 & 등록
        - @ComponentScan ⇒ @Component, @Service, @Repository, @Controller 애노테이션이 붙은 클래스 탐색 및 빈 등록
        - 수동 등록 ⇒ @Bean
    - 의존 관계 설정

## 5장

- 스프링 : 싱글톤 컨테이너
- 싱글톤 컨테이너 : 객체 인스턴스 싱글톤(1개만 생성) 관리
- 스프링 빈 : 싱글톤으로 관리되는 빈
- 싱글톤 방식의 주의점 : 무상태(stateless)로 설계해야 함
- 스프링 설정 정보 → @Configuration 사용

## 6장

- 컴포넌트 스캔 : 설정 정보가 없어도 자동으로 스프링 빈 등록
    - @ComponentScan → 이걸 설정 정보에 붙여주면 됨
    - @Configuration → 이거 붙어있는 설정 정보는 자동 등록됨
        - 제외하고 싶으면 excludeFilters 사용

## 7장

- 생성자 주입
    - 객체 생성 시 한 번만 호출 → 변경 x
    - 생성자가 하나면 @autowired 생략해도 자동 주입 됨
- 수정자 주입
    - 필드의 값을 변경하는 메서드를 통해 의존관계를 주입
- 조회 빈이 2개 이상일 때 문제 해결
    - nonuniquebeandefinitionexception 오류 발생
    - 해결 방법
        - @autowired 필드명 매칭
        - @qualifier 추가 구분자 입력
        - @primary 우선권 부여

## 8장

- 빈 생명주기 콜백 : 빈의 초기화, 소멸 전에 자동으로 호출되는 메서드. 개발자에게 빈 사용 시기를 안내해줌
- @PostConstruct(초기화 메서드 위에), @PreDestroy(종료 메서드 위에) 애노테이션 사용
    - 외부 라이브러리에선 코드를 고칠 수 없어서 사용이 어려움 →@Bean의 initMethod, destroyMethod 속성 사용

## 9장

- 싱글톤 스코프
    - 스프링 빈의 기본 스코프
    - 스프링 컨테이너가 시작될 때 딱 하나만 만들어져서 컨테이너가 종료될 때까지 계속 유지
    - 스프링 컨테이너가 시작부터 끝까지 책임
- 프로토타입 스코프
    - 요청할 때마다 새로운 빈 생성
    - 스프링 컨테이너가 생성, 의존관계 주입, 초기화까지만 처리
- 웹 스코프
