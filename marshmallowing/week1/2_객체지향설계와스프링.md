### 섹션2. 객체 지향 설계와 스프링

### 스프링

- 자바(객체 지향 언어) 언어 기반의 프레임워크
  ⇒ 좋은 **객체 지향 애플리케이션**을 개발할 수 있게 도와주는 프레임워크

### 좋은 객체 지향 프로그래밍

- **다형성**
    - ***역할과 구현의 분리*** ⇒ 유연과 변경 편리
    - **역할 = 인터페이스/구현=인터페이스를 구현한 클래스, 구현 객체**
    - 객체 설계 시 역할(인터페이스)을 먼저 부여하고, 그 역할을 수행하는 구현 객체 만듦
    - 클라이언트는 구현 대상 자체를 변경해도 영향 받지 않음, 대상의 역할 (인터페이스)에만 영향 받음
    - (+) 오버라이딩 vs 오버로딩
        - **오버로딩과 오버라이딩 비교**
            - 같은 클래스에서 같은 이름, 매개변수, 타입, 개수가 다르면 오버로딩
            - 오버라이딩은 상속에서 이루어지는 것 (기본 클래스와 파생 클래스 사이에) 리턴 값과 매개변수, 타입 등이 모두 동일
            - 다형성- 실행 중에 어떤 걸 수행할지 판단

    💡
    인터페이스를 구현한 **객체 인터페이스**를 실행 시점에 유연하게 변경 가능 & 클라이언트를 변경하지 않고, 서버의 구현 기능을 유연하게 변경 가능
    </aside>

### **좋은 객체 지향 설계의 5가지 원칙 (SOLID)**

- **SRP 단일 책임 원칙**
    - 한 클래스는 하나의 책임만 가져야 한다.
- **OCP 개방-폐쇄 원칙**
    - 확장에는 열려 있으나 변경에는 닫혀 있어야 한다

    ```java
    public class MemberService {
    // private MemberRepository memberRepository = new MemoryMemberRepository();
     private MemberRepository memberRepository = new JdbcMemberRepository();
    }
    ```

    - 이건 `MemberService`의 코드 자체를 변경하는 것으로 안 좋은 방법
      ⇒ OCP 깨짐 (클라이언트가 코드를 변경해야 함)
      → 객체를 생성하고, 연관관계를 맺어주는 별도의 조립, 설정자 필요 (스프링이 해줌, 이후에 다룸)

---

- **LSP 리스코프 치환 원칙**
    - 프로그램의 객체는 프로그램의 정확성을 깨뜨리지 않으면서 하위 타입의 인스턴스로 바꿀 수 있어야 한다.
    - 다형성에서 하위 클래스는 인터페이스 규약을 다 지켜야 한다는 의미
      (ex. 자동차 인터페이스의 엑셀은 앞으로 가라는 기능, 뒤로 가게 구현 시 LSP 위반하는 것)

---

- **ISP 인터페이스 분리 원칙**
    - 특정 클라이언트를 위한 인터페이스 여러 개가 범용 인터페이스 하나보다 낫다
    - 인터페이스 분리 시, 인터페이스가 명확해지고 대체 가능성 높아짐

---

- **DIP 의존 관계 역전 원칙**
    - 프로그래머는 추상화에 의존해야지 구체화에 의존하면 안 된다.
    - **구현 클래스에 의존하지 말고, 인터페이스에만 의존**

  ⇒ 유연하게 구현체 변경 가능

    ```java
    public class MemberService {
    // private MemberRepository memberRepository = new MemoryMemberRepository();
     private MemberRepository memberRepository = new JdbcMemberRepository();
    }
    ```

    - `MemberService`는 인터페이스에 의존하지만, 구현 클래스도 동시에 의존한다
      (memberRepository:인터페이스 / MemoryMemberRepository: 구현 클래스)
    - 알기만 해도 의존한다고 여긴다 (구현 클래스까지 알고 있음)
    - `MemberService` 클라이언트가 구현 클래스를 직접 선택하고 있음

  ***⇒ DIP 위반***
