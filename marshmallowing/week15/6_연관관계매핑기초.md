### 객체와 테이블이 관계를 맺는 차이

- **객체 연관관계 = 2개**
    - 회원 → 팀 연관관계 1개 (단방향)
    - 팀 → 회원 연관관계 1개 (단방향)
    - 참조 값이 양쪽에 각각 존재해야 함 → 사실상 단방향 2개
- **테이블 연관관계 = 1개**
    - 회원 ↔ 팀의 연관관계 1개(양방향)
    - TEAM_ID 하나만으로 양쪽에서 서로를 찾을 수 있음

### 객체의 양방향 관계 ⭐

- **객체의 양방향 관계는 사실 양방향 관계가 아니라 서로 다른 단방향 관계 2개이다**
- 객체를 양방향으로 참조하려면 **단방향 연관관계를 2개** 만들어야 한다

### 테이블의 양방향 연관관계

- 테이블은 **외래 키 하나**로 두 테이블의 연관관계를 관리
- MEMBER.TEAM_ID 외래 키 하나로 양방향 연관관계 가짐
    - 양 쪽으로 조인할 수 있다

```sql
SELECT * 
FROM MEMBER M
JOIN TEAM T ON M.TEAM_ID = T.TEAM_ID
-----
SELECT *
FROM TEAM T
JOIN MEMBER M ON T.TEAM_ID = M.TEAM_ID
```

### 연관관계의 주인(Owner)

### 둘 중 하나로 외래 키를 관리해야 한다

![image.png](attachment:49345c40-5a51-482e-bbe1-b47e75c4da87:image.png)

members와 team을 독립적으로 조작할 경우 생길 오류들 어떻게? (한 쪽에만 값 넣고 ..)

### **양방향 매핑 규칙**

- 객체의 두 관계 중 하나를 연관관계의 주인으로 지정
- **연관관계의 주인만이 외래 키를 관리(등록, 수정)**
- **주인이 아닌쪽은 읽기만 가능**
- 주인은 mappedBy 속성 사용 X
    - 주인이 아니면 mappedBy 속성으로 주인 지정

### 누구를 주인으로?

- **외래 키가 있는 곳을 주인으로 정해라(N쪽)**
- 여기서는 **Member.team**이 연관관계의 주인

![image.png](attachment:3111d66b-6768-4baa-a5e7-450a38ca0280:image.png)

```java
@OneToMany(**mappedBy = "team"**) //team에 의해 관리됨
private List<Member> members = new ArrayList<>();
```

*→ members에 값 넣어도 아무일도 일어나지X, 읽기, 조회만 가능*

- `@OneToMany(mappedBy = "tag", cascade = CascadeType.*ALL*, orphanRemoval = true)`
    - 영속성 전이 cascade
        - 부모 엔티티의 생명주기에 종속적인 자식 엔티티의 상태변화를 같이 처리해주는 역할
    - 고아 객체 orphanRemoval
        - 연관 관계가 사라진, 고아가 된 entity를 삭제하는 것
        - `CascadeType.REMOVE`와 달리  관계만 제거해도 자동 삭제됨

```java
Team team = new Team();
team.setName("TeamA");
em.persist(team);

Member member = new Member();
member.setName("member1");

//역방향(주인이 아닌 방향)만 연관관계 설정
team.getMembers().add(member); //null
em.persist(member);
```

```java
Team team = new Team();
team.setName("TeamA");
em.persist(team);

Member member = new Member();
member.setName("member1");

**//1차캐시에 넣은 상태에서(디비 반영전) 이 코드 없을 시 연관관계 형성 안됨**
**team.getMembers().add(member);**
//**연관관계의 주인에 값 설정
member.setTeam(team);**
/*위 두 코드 반드시 필요*/

em.persist(member);

em.flush(); // 1차 캐시 → DB에 INSERT
em.clear(); // 1차 캐시 비움

for(Member m : members){
	System.out.printlm(m);
}
```

| 상황 | `member.setTeam(team)`만 한 경우 | `team.getMembers().add(member)`만 한 경우 |
| --- | --- | --- |
| 1차 캐시(트랜잭션 중)에서 조회 | `team.getMembers()` 비어 있음 | `member.getTeam()` = null |
| flush/clear 후 DB에서 조회 | `team.getMembers()` OK | 연관관계가 DB에 안 들어감 (`team_id` null) |

⇒ 트랜잭션 중 1차 캐시에서 컬렉션 동기화하려면 **양쪽 다 set()과 add()를 해줘야** 한다

### 양방향 연관관계 주의

- **순수 객체 상태를 고려해서 항상 양쪽에 값을 설정하자**
- **연관관계 편의 메소드**를 생성하자
    - 하나만 호출해도 양쪽에 세팅 가능

    ```java
    public void setTeam(Team team) {
    		this.team = team;
    		team.getMembers().add(this);
    }
    ```

    ```java
    public void addMember(Member member) {
    		member.setTeam(this);
    		members.add(member);
    }
    ```

    - 상황에 따라 둘 중 하나 선택
- 양방향 매핑 시에 무한 루프를 조심하자
    - toString(), lombok, JSON 생성 라이브러리
    - controller에서는 엔티티 반환 금지!!
        - 무한 루프나 엔티티 변경 위험 / dto로 반환하기

### 양방향 매핑 정리

- **단방향 매핑만으로 이미 연관관계 매핑은 완료**
    - 다에다 세팅
- **양방향 매핑(mapped by)은 반대 방향으로 조회(객체 그래프 탐색) 기능이 추가된 것 뿐**
- JPQL에서 역방향으로 탐색할 일이 많음
- 단방향 매핑을 잘 하고 양방향은 필요할 때 추가해도 됨
    - 테이블에 영향을 주지 않음
    - `설계 시 단방향만 세팅하고 실제 개발하다 필요 시 양방향 세팅하기`

### 연관관계의 주인을 정하는 기준

- 비지니스 로직을 기준으로 연관관계의 주인을 선택하면 안됨
- **연관관계의 주인은 외래 키의 위치를 기준으로 정해야함**

### 연관관계 매핑 시작

![테이블 구조](attachment:4246b3c0-82b9-47ab-8daa-780bfd2c47d1:image.png)

테이블 구조

![객체 구조](attachment:1acbacc6-6a2f-4e17-932e-b8900e069142:image.png)

객체 구조

| 엔티티 | 관계 | 설명 |
| --- | --- | --- |
| `Order` ↔ `OrderItem` | 1:N | 하나의 주문에 여러 주문상품 (OrderItem)이 들어감 |
| `OrderItem` ↔ `Item` | N:1 | 여러 주문상품이 하나의 상품(Item)을 참조할 수 있음 |

```java
@Entity
public class Order {

    @Id @GeneratedValue
    private Long id;

    @OneToMany(mappedBy = "order")
    private List<OrderItem> orderItems = new ArrayList<>();
}
```

- 하나의 `Order`는 여러 `OrderItem`을 가짐 → **1:N 관계**

---

```java
@Entity
public class OrderItem {

    @Id @GeneratedValue
    private Long id;

    @ManyToOne
    @JoinColumn(name = "order_id")
    private Order order;

    @ManyToOne
    @JoinColumn(name = "item_id")
    private Item item;
}
```

- `OrderItem`은 **하나의 `Order`에 속함** → `@ManyToOne`
- `OrderItem`은 **하나의 `Item`을 참조함** → `@ManyToOne`

---

```java
@Entity
public class Item {

    @Id @GeneratedValue
    private Long id;

    private String name;
}
```

- Item은 자신이 어떤 주문에 쓰였는지 알 필요가 없으므로, **양방향 관계 X (단방향)**
